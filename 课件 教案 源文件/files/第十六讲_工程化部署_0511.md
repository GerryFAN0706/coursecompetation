## 统一异常处理

###### Unified Exception Handling

**问题 (Problem)**:
- 每个Servlet都要try-catch
- 异常显示不友好（白屏、堆栈）
- 难以统一管理

**解决方案 (Solution)**:
- 全局异常过滤器
- 统一错误页面

**GlobalExceptionFilter**:
```java
@WebFilter("/*")
public class GlobalExceptionFilter implements Filter {
    public void doFilter(...) {
        try {
            chain.doFilter(req, res);
        } catch (Throwable ex) {
            // 记录日志
            ex.printStackTrace();
            // 跳转错误页
            response.sendRedirect("error.jsp");
        }
    }
}
```

## 异常处理流程

###### Exception Handling Flow

**两级异常处理机制 (Two-Level Exception Handling)**:

```
客户端请求 (Client Request)
  ↓
Servlet抛出异常 (Servlet throws Exception)
  ↓
┌─────────────────────────────────┐
│ 第一级: Filter捕获              │
│  GlobalExceptionFilter          │
│  ├── 记录日志 (Log exception)    │
│  ├── 设置错误属性                │
│  └── 转发错误页 (Forward to JSP) │
└─────────────────────────────────┘
  ↓ (未被Filter捕获)
┌─────────────────────────────────┐
│ 第二级: 容器error-page           │
│  web.xml <error-page>           │
│  ├── 404 → /jsp/404.jsp         │
│  ├── 500 → /jsp/500.jsp         │
│  └── 403 → /jsp/403.jsp         │
└─────────────────────────────────┘
  ↓
用户看到友好错误页 (Friendly Error Page)
```

**设计原则**: Filter优先处理，error-page作为兜底
*Filter handles first, error-page as fallback*

## 全局错误页配置

###### Global Error Page Configuration

**web.xml配置 (Configuration)**:

```xml
<error-page>
    <error-code>404</error-code>
    <location>/jsp/404.jsp</location>
</error-page>

<error-page>
```

## 统一日志记录

###### Unified Logging

**日志过滤器 (Logging Filter)**:

```java
@WebFilter("/*")
public class LoggingFilter implements Filter {
    public void doFilter(...) {
        long start = System.currentTimeMillis();
        String uri = request.getRequestURI();
        
        System.out.println("[ACCESS] " + uri);
        
        try {
            chain.doFilter(req, res);
        } finally {
            long cost = System.currentTimeMillis() - start;
            System.out.println("[ACCESS] " + uri +
                             " cost=" + cost + "ms");
        }
    }
}
```

## 日志最佳实践

###### Logging Best Practices

**关键日志信息 (Essential Log Fields)**:

| 字段 | 说明 | 示例 |
|-----|------|-----|
| **TraceID** | 请求追踪标识 | `TRC-20251205-001` |
| **用户标识** | 当前操作用户 | `user=zhangsan` |
| **URI** | 请求路径 | `/api/student/list` |
| **耗时** | 处理时间 | `cost=125ms` |
| **异常堆栈** | 错误详情 | `NullPointerException` |

**日志级别策略 (Log Level Strategy)**:

<!-- _class: cols2_ol_ci -->

- **DEBUG** 开发环境详细信息
  *Development details*
- **INFO** 生产环境关键操作
  *Production key actions*
- **WARN** 告警但不影响运行
  *Warnings, non-critical*
- **ERROR** 异常需要处理
  *Exceptions to handle*

**生产建议**: 接入 **SLF4J + Logback** 框架，替代System.out
*Production: Use SLF4J + Logback instead of System.out*

## 为什么需要配置分离？

###### Why Configuration Separation?

**硬编码的问题 (Hardcoded Problems)**:

<!-- _class: cols2_ol_ci -->

- **安全隐患** 密码在代码中  
  *Security risk*
- **无法切换** 开发/测试/生产环境  
  *Can't switch env*
- **修改麻烦** 改配置要重新编译  
  *Hard to modify*
- **团队协作** 每人配置不同  
  *Team issues*

**JNDI方案 (JNDI Solution)**:
> 将数据源配置到Tomcat容器  
> 应用通过名称查找使用  
> *Configure in container, lookup by name*

## JNDI数据源配置

###### JNDI DataSource Configuration

**Tomcat配置 (context.xml)**:

```xml
<Resource 
    name="jdbc/myDB" 
    auth="Container"
    type="javax.sql.DataSource"
    driverClassName="com.mysql.cj.jdbc.Driver"
    url="jdbc:mysql://localhost:3306/demo"
    username="root"
    password="123456"
    maxTotal="20"
    maxIdle="5"
/>
```

**应用使用 (Application Usage)**:
```java
Context ctx = new InitialContext();
DataSource ds = (DataSource) ctx.lookup("java:comp/env/jdbc/myDB");
Connection conn = ds.getConnection();
```

## 多环境配置策略

###### Multi-Environment Configuration Strategy

**环境差异对比 (Environment Comparison)**:

| 配置项 | 开发环境 | 测试环境 | 生产环境 |
|-------|---------|---------|---------|
| **数据库** | 本地MySQL | 测试服务器 | 主从集群 |
| **maxTotal** | 5 | 10 | 50 |
| **SSL连接** | 否 | 否 | 是 |
| **日志级别** | DEBUG | INFO | WARN |
| **密码管理** | 本地配置 | 环境变量 | 密钥管理服务 |

**配置分离原则 (Separation Principle)**:
> 代码中不含任何环境参数
> 每个环境独立context.xml
> 敏感信息通过环境变量注入
> *Code contains ZERO environment parameters*

## 安全配置检查清单

###### Security Configuration Checklist

**部署前安全审查 (Pre-Deployment Security Review)**:

<!-- _class: cols2_ol_ci -->

- **密码管理** 不能明文存储
  *No plaintext passwords*
- **数据库连接** 使用SSL加密
  *Use SSL for DB connections*
- **连接池参数** 合理设置上限
  *Reasonable pool limits*
- **敏感配置** 不进代码仓库
  *Secrets out of repo*
- **管理端口** 修改默认端口
  *Change default ports*
- **默认凭据** 必须修改
  *Change default credentials*

**AI安全审计实践 (AI Security Audit)**:

<div class="ai-box">

**Cursor安全检查**: 让AI审查context.xml，识别明文密码、默认端口、未加密连接等问题。学生对照检查清单验证AI建议的准确性。

## WAR打包

###### WAR Packaging

**什么是WAR？** *What is WAR?*

**Web Application Archive** - Web应用归档文件

**打包方式 (Packaging Methods)**:

<!-- _class: cols2_ol_ci -->

- **IDE打包** IntelliJ/Eclipse  
  *IDE packaging*
- **Maven打包** mvn package  
  *Maven build*
- **Gradle打包** gradle war  
  *Gradle build*
- **手动打包** jar -cvf  
  *Manual packaging*

**WAR结构 (WAR Structure)**:
```
myapp.war
  ├── WEB-INF/
  │   ├── classes/    ← Java类
  │   ├── lib/        ← JAR包
  │   └── web.xml
  └── jsp/            ← JSP文件
```

## Tomcat部署

###### Tomcat Deployment

**部署步骤 (Deployment Steps)**:

```
1. 打包WAR文件
   mvn clean package → myapp.war
   ↓
2. 停止Tomcat
   bin/shutdown.bat
   ↓
3. 复制WAR到webapps/
   copy myapp.war tomcat/webapps/
   ↓
4. 启动Tomcat
   bin/startup.bat
   ↓
5. Tomcat自动解压WAR
   ↓
6. 访问测试
   http://localhost:8080/myapp/
```

## 部署验证清单

###### Deployment Verification Checklist

**上线后必检项目 (Post-Deployment Checks)**:

<!-- _class: cols2_ol_ci -->

- **启动日志** 无ERROR/WARN
  *Clean startup logs*
- **首页访问** HTTP 200正常
  *Homepage returns 200*
- **登录功能** 用户认证正常
  *Login authentication works*
- **数据库连接** CRUD操作正常
  *DB CRUD operations OK*
- **静态资源** CSS/JS/图片加载
  *Static resources load*
- **会话管理** Session有效
  *Session management works*

**验证命令 (Verification Commands)**:
```bash
# 检查Tomcat进程
ps -ef | grep tomcat
# 检查启动日志
tail -100 logs/catalina.out | grep ERROR
# 测试HTTP响应
curl -I http://localhost:8080/myapp/
```

## 故障排查指南

###### Troubleshooting Guide

**常见部署问题与解决方案 (Common Issues)**:

| 错误现象 | 可能原因 | 排查方法 |
|---------|---------|---------|
| **404** | WAR名称与URL不匹配 | 检查webapps/目录名 |
| **500** | 代码异常或配置错误 | 查看catalina.out日志 |
| **连接超时** | 数据库配置或防火墙 | telnet检查端口连通性 |
| **ClassNotFound** | 依赖JAR缺失 | 检查WEB-INF/lib/ |

**排查步骤 (Debugging Steps)**:
```
1. 查看日志 → tail -f logs/catalina.out
2. 检查配置 → context.xml / web.xml
3. 验证依赖 → WEB-INF/lib/ 是否完整
4. 测试连接 → 数据库/网络是否可达
5. 对比环境 → 开发环境能否复现
```
