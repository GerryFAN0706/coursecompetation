老师好，我想问一下Session和Cookie到底有什么区别？什么时候该用哪个？
20:17
W
这是个很好的问题！在 第5讲 会话管理 中我们详细讲解了这个知识点，我帮你梳理一下核心区别：

1. 存储位置不同
• Cookie 存储在客户端浏览器中
• Session 存储在服务器端（默认在内存中）

2. 安全性差异
• Cookie 可被用户查看和篡改，安全性较低
• Session 数据在服务器端，相对安全

3. 生命周期
• Cookie 可设置过期时间，持久存储
• Session 默认随浏览器关闭而失效（Tomcat默认30分钟超时）

4. 典型使用场景
• Cookie：记住用户名、语言偏好等非敏感信息
• Session：登录状态、购物车、验证码等敏感数据

// 第5讲 代码示例：Session存储登录信息 HttpSession session = request.getSession(); session.setAttribute("currentUser", user); // Cookie记住用户偏好 Cookie cookie = new Cookie("theme", "dark"); cookie.setMaxAge(7 * 24 * 3600); response.addCookie(cookie);
记忆口诀：敏感数据用Session，偏好记忆用Cookie。面试中也经常考到这个对比哦！

那Session的JSESSIONID和Cookie是什么关系？
20:18
W
问得好！这正是 第5讲 的一个重点，也是很多同学容易混淆的地方：

Session 依赖 Cookie 来工作！具体流程是：

• 服务器创建Session后，生成一个唯一的 JSESSIONID
• 这个ID通过 Cookie 发送给浏览器（名为JSESSIONID的Cookie）
• 浏览器后续请求自动携带这个Cookie
• 服务器根据JSESSIONID找到对应的Session对象

所以本质上：Session是机制，Cookie是载体。如果浏览器禁用了Cookie，可以用URL重写方式传递JSESSIONID。

这部分在期末考试中占比较高，建议结合 第5讲 实验 用Servlet代码实际调试一下！

Filter的doFilter方法中chain.doFilter()的作用是什么？
20:22
W
这是 第9讲 Filter/Listener 的核心知识点！

chain.doFilter(request, response) 的作用是将请求传递给过滤器链中的下一个Filter，如果当前是最后一个Filter，则传递给目标Servlet。

关键理解：
• 调用 chain.doFilter() = 放行请求，继续执行后续处理
• 不调用 chain.doFilter() = 拦截请求，后续的Filter和Servlet都不会执行
• chain.doFilter() 前后的代码分别对应请求前处理和响应后处理

public void doFilter(ServletRequest req, ServletResponse resp, FilterChain chain) throws IOException, ServletException { // ---- 请求前处理（Pre-processing）---- System.out.println("请求到达前的逻辑"); long start = System.currentTimeMillis(); // ---- 放行请求 ---- chain.doFilter(req, resp); // ---- 响应后处理（Post-processing）---- long cost = System.currentTimeMillis() - start; System.out.println("请求耗时: " + cost + "ms"); }
这就像一个洋葱模型：请求进入时从外到内经过每个Filter，响应返回时从内到外再经过每个Filter。理解了这一点，Filter链的执行顺序就清楚了！

MVC架构中Controller和Service层的区别？
20:30
W
好问题！这涉及到 第10讲 MVC设计模式 和 第11讲 Spring MVC 的核心分层思想：

Controller层（控制层）
• 负责接收用户请求和返回响应
• 进行请求参数的解析与校验
• 调用Service层完成业务逻辑
• 选择合适的视图返回给用户
• 不包含业务逻辑代码

Service层（业务逻辑层）
• 封装核心业务逻辑
• 调用DAO层完成数据操作
• 处理事务管理
• 可被多个Controller复用
• 不关心HTTP请求/响应细节

// Controller层 - 只做请求调度 @Controller public class UserController { @Autowired private UserService userService; @PostMapping("/login") public String login(String username, String password, HttpSession session) { User user = userService.authenticate(username, password); if (user != null) { session.setAttribute("currentUser", user); return "redirect:/home"; } return "login"; } } // Service层 - 封装业务逻辑 @Service public class UserService { @Autowired private UserDAO userDAO; public User authenticate(String username, String pwd) { User user = userDAO.findByUsername(username); if (user != null && BCrypt.check(pwd, user.getPassword())) { return user; } return null; } }
一句话总结：Controller是"调度员"（接客），Service是"厨师"（做菜），DAO是"仓库管理员"（取食材）。各司其职才能让代码清晰、可维护！

PreparedStatement为什么能防止SQL注入？
20:38
W
这是 第8讲 JDBC编程 和 第13讲 Web安全基础 的交叉重点，也是面试必考题！

核心原因：PreparedStatement 将 SQL语句结构 和 参数数据 分离

使用Statement（不安全）：
// 字符串拼接 - 存在SQL注入风险！ String sql = "SELECT * FROM users WHERE name='" + username + "' AND pwd='" + password + "'"; // 恶意输入: username = admin' OR '1'='1' -- // 拼接后: SELECT * FROM users WHERE name='admin' OR '1'='1' --' AND pwd='' // 结果: 绕过密码验证！
使用PreparedStatement（安全）：
// 参数化查询 - 防止SQL注入 String sql = "SELECT * FROM users WHERE name=? AND pwd=?"; PreparedStatement ps = conn.prepareStatement(sql); ps.setString(1, username); // 自动转义特殊字符 ps.setString(2, password); // 恶意输入被当作纯文本处理，不会改变SQL结构
防御机制详解：
• SQL语句在 prepareStatement() 时预编译，结构已固定
• setString() 会自动对特殊字符（如单引号）进行转义处理
• 参数值只能作为数据，不能作为SQL关键字或语法的一部分

总结：预编译 + 参数绑定 = 结构与数据分离 = 从根本上杜绝注入。这个知识点在第8讲JDBC实验中可以动手验证！

如何实现用户登录状态的保持？
20:45
W
这是 第5讲 会话管理 的实战应用场景，我来给你梳理完整的实现方案：

方案一：Session（服务端方案，最常用）
// 登录成功后，将用户信息存入Session HttpSession session = request.getSession(); session.setAttribute("currentUser", user); session.setMaxInactiveInterval(30 * 60); // 30分钟超时 // 判断是否已登录 Object user = session.getAttribute("currentUser"); if (user == null) { response.sendRedirect("/login.jsp"); }
方案二：Cookie + Session（记住我功能）
• 用户勾选"记住我"时，生成一个加密Token存入Cookie
• Cookie设置较长的过期时间（如7天）
• 下次访问时，Filter检查Cookie中的Token，自动恢复Session

方案三：登录拦截Filter（全局保护）
// LoginFilter - 统一拦截未登录访问 public void doFilter(...) { HttpServletRequest req = (HttpServletRequest) request; String uri = req.getRequestURI(); // 放行登录页和静态资源 if (uri.contains("/login") || uri.contains("/static")) { chain.doFilter(request, response); return; } // 检查Session中的登录状态 HttpSession session = req.getSession(false); if (session != null && session.getAttribute("currentUser") != null) { chain.doFilter(request, response); } else { ((HttpServletResponse)response).sendRedirect("/login.jsp"); } }
这三个方案在 第15讲 项目实战(上) 中有完整代码实现，建议对照实验动手练习！
