# 第十五讲课堂实录逐分钟讲稿

> 课程：`Java Web应用开发`  
> 赛道：`人工智能赛道`  
> 课堂主题：`从“能登录”到“能守住后台”：用户认证与权限控制`  
> 建议时长：`45分钟`

## 一、课堂定位

本节课不按“知识点平铺讲授”展开，而按一个真实问题展开：

## `为什么“已经实现了登录”仍然不等于“系统是安全的”？`

课堂围绕一个存在缺陷的 Java Web 项目展开，学生借助 AI 助教完成：

1. 问题识别
2. 知识点对照
3. 代码修改
4. 结果验证
5. 复盘总结

## 二、课堂目标

### 知识目标

1. 理解 `Authentication` 与 `Authorization` 的区别。
2. 理解 `Session-Based Login` 的基本机制。
3. 理解 `RBAC` 在 Java Web 中的基本实现方式。
4. 理解 `Filter` 在权限控制中的作用。

### 能力目标

1. 能分析登录模块中的认证与授权缺陷。
2. 能借助 AI 助教进行代码审查和知识点检查。
3. 能修改登录、过滤器、视图层相关代码。
4. 能通过不同角色路径验证系统行为。

### 素质目标

1. 建立“默认拒绝、最小权限”的安全意识。
2. 理解 AI 是辅助学习与诊断的工具，不是代替思考的工具。

## 三、课前准备

上课前建议准备好以下内容：

1. 一个可运行的 Java Web 项目
2. 两个账号
   - `admin / ADMIN`
   - `tom / USER`
3. 一份“故意留有权限漏洞”的代码版本
4. 三组任务单
5. AI 提示词卡片
6. 浏览器已可访问 `/login.jsp`、`/admin/panel`
7. 投影能展示：
   - 代码
   - 浏览器运行结果
   - AI 对话界面

## 四、课堂分组安排

建议分为 3 组：

1. `A组：认证组`
   - 聚焦 `LoginServlet`、`LogoutServlet`
2. `B组：授权组`
   - 聚焦 `RoleBasedAuthFilter`
3. `C组：视图一致性组`
   - 聚焦 JSP 菜单渲染与角色显示逻辑

## 五、45分钟逐分钟讲稿

## 0-2分钟：开场与任务抛出

### 教师讲什么

“同学们，前面我们已经会做登录功能了。但今天我要带大家解决一个更真实的问题：系统能登录，不代表系统是安全的。一个普通用户，如果也能进入后台，那这个系统就只是‘看起来能用’，实际上并不可靠。”

“今天这节课，我们不只学概念，而是要一起修复一个带有安全缺陷的 Java Web 系统。”

### 教师操作

1. 打开项目首页
2. 简要展示登录页与后台页入口

### 学生活动

1. 进入课堂状态
2. 观察系统界面

### 板书/投影

投影标题：

`Can Login != Can Securely Control Access`

## 2-5分钟：演示问题系统，制造认知冲突

### 教师讲什么

“请大家观察三个问题：

1. 未登录用户能否访问后台？
2. 普通用户登录后能否访问后台？
3. 菜单显示是否与权限控制一致？”

“如果答案处理不好，就会出现‘会登录、但不安全’的问题。”

### 教师操作

现场演示 1 次错误行为，例如：

1. `tom / USER` 登录
2. 直接访问 `/admin/panel`
3. 发现系统仍能访问，或菜单显示异常，或权限判断不完整

### 教师追问

“这个问题属于认证错误，还是授权错误？”

### 学生可能回答

1. “是授权问题。”
2. “用户已经登录了，但不代表有权限进入后台。”

### 教师点拨

“很好。认证回答的是‘你是谁’，授权回答的是‘你能做什么’。”

## 5-8分钟：明确本课目标与分组任务

### 教师讲什么

“今天我们用 AI 助教做 3 件事：

1. 帮我们快速找到代码中的风险点
2. 对照本节课知识点检查问题归属
3. 帮我们生成验证清单

但注意，AI 不替你写答案，最后必须由你自己判断、修改和验证。”

### 教师操作

展示三组任务：

1. A组：检查登录与登出是否规范
2. B组：检查角色权限过滤是否正确
3. C组：检查 JSP 菜单显示是否与服务端控制一致

### 学生活动

1. 按组就位
2. 打开代码与 AI 工具

### 板书/投影

`Authentication / Authorization / Session / RBAC / Filter`

## 8-10分钟：知识点快速唤醒

### 教师讲什么

“在开始之前，请大家快速回忆：

1. 登录成功后，用户信息通常保存在什么地方？
2. 保护 `/admin/*` 这类路径，最适合放在哪一层做？
3. 前端把菜单隐藏掉，是否就等于安全？”

### 学生回答

预期答案：

1. Session
2. Filter
3. 不等于，后端必须控制

### 教师总结

“这三个点，正是我们今天课堂要修复的关键。”

## 10-13分钟：教师示范一次 AI 代码审查

### 教师讲什么

“我先演示一次，AI 助教如何帮助我们做诊断。注意它输出的是‘候选意见’，不是‘标准答案’。”

### 教师操作

选中一段与权限控制相关的代码，例如 `RoleBasedAuthFilter`，输入提示词：

```text
请审查这个 Java Web 用户认证与权限控制模块，重点检查：
1. 认证流程是否正确
2. Session 的创建、保存、注销是否规范
3. 是否存在未授权访问或越权访问风险
4. RoleBasedAuthFilter 是否真正实现了基于角色的控制
5. JSP 菜单渲染是否与服务端权限控制一致
6. 密码处理是否存在安全隐患

请按以下格式输出：
- 问题位置
- 风险说明
- 对应课程知识点
- 修改建议
- 建议验证方式
```

### 教师边演示边说

“大家注意我最关心的不是 AI 给出什么代码，而是它是否能：

1. 指出问题位置
2. 说明风险
3. 对应课程知识点
4. 提出可验证建议”

### 学生活动

观看演示，准备本组操作

## 13-20分钟：学生分组使用 AI 审查代码

### 教师讲什么

“现在各组开始。你们的任务不是抄 AI 的答案，而是判断：AI 说的到底对不对，为什么对，为什么不完全对。”

### A组任务

聚焦：

1. `LoginServlet`
2. `LogoutServlet`

关注点：

1. 登录后是否正确保存用户到 Session
2. 登出是否真正清除登录状态
3. 是否存在会话残留问题

### B组任务

聚焦：

1. `RoleBasedAuthFilter`

关注点：

1. 是否只判断“登录”
2. 是否正确判断 `ADMIN`
3. 普通用户是否仍可进入 `/admin/*`

### C组任务

聚焦：

1. JSP 页面菜单
2. `sessionScope.user`

关注点：

1. 菜单入口是否按角色显示
2. 前端显示与后端权限是否一致
3. 是否有“隐藏了入口但仍可直接访问”的问题

### 教师巡视时可说的话

1. “不要只问 AI ‘怎么改’，先问‘哪里有风险’。”
2. “把 AI 输出和我们今天的知识点对应起来。”
3. “每组最后要说清楚：发现了什么问题，对应什么知识点。”

### 学生活动

1. 输入提示词
2. 阅读 AI 输出
3. 圈出本组最关键的 1-2 个问题
4. 准备汇报

### 视频重点

必须拍到：

1. 学生实际输入提示词
2. AI 返回结果
3. 学生讨论结果

## 20-26分钟：小组汇报 AI 诊断结果

### 教师讲什么

“每组只汇报三件事：

1. AI 找到了什么问题
2. 它对应什么知识点
3. 哪条建议你们认为最值得采纳”

### A组预期汇报

可能内容：

1. 登录成功后需要把用户对象放到 Session
2. 注销必须 `invalidate`
3. 如果只移除部分属性而不失效 Session，可能造成状态残留

### B组预期汇报

可能内容：

1. Filter 只检查是否登录还不够
2. 必须进一步判断角色是否为 `ADMIN`
3. 普通用户访问后台属于越权问题

### C组预期汇报

可能内容：

1. 菜单按角色显示只是界面控制
2. 真正安全要靠服务端过滤器
3. 视图层与服务端必须一致

### 教师点评讲稿

“A组抓住了会话管理，B组抓住了授权控制，C组抓住了一个很容易被忽略的问题：前端隐藏按钮不等于真正安全。也就是说，认证、授权、视图一致性是一个整体。”

## 26-30分钟：教师系统提升与知识点对齐

### 教师讲什么

“现在我们把今天的问题和课程知识点一一对齐。”

可边讲边写：

1. `Authentication`
   - 登录是否成功
2. `Authorization`
   - 登录后能访问哪些资源
3. `Session-Based Login`
   - 用户状态如何跨请求保存
4. `RBAC`
   - 不同角色拥有不同访问权限
5. `Filter`
   - 对受保护路径统一做控制

### 教师点睛语

“如果只会登录，不会授权，你做出来的是‘能运行的系统’，不是‘可信的系统’。”

## 30-35分钟：学生修改关键代码

### 教师讲什么

“下面进入修复阶段。请每组只改与你们最相关的关键点，不追求把整个项目全改完，先确保核心行为正确。”

### 学生活动

#### A组修改建议

1. 登录成功后：
   - `req.getSession().setAttribute("user", u);`
2. 登出：
   - `session.invalidate();`

#### B组修改建议

1. 在 `RoleBasedAuthFilter` 中增加角色判断
2. 仅 `ADMIN` 放行

#### C组修改建议

1. JSP 中按角色动态渲染菜单
2. 保证菜单与服务端规则一致

### 教师巡视引导语

1. “先改最关键逻辑，再考虑代码美观。”
2. “修改后一定要验证，不能只看代码感觉没问题。”

## 35-39分钟：借助 AI 生成测试清单并验证

### 教师讲什么

“软件安全不是靠感觉，必须靠验证。下面请大家让 AI 助教给出最小验证清单，然后现场测试。”

### 教师示范提示词

```text
请为这个 Java Web 登录与权限控制模块生成一个最小验证清单，覆盖：
1. 未登录访问后台
2. 普通用户访问后台
3. 管理员访问后台
4. 注销后再次访问后台
5. JSP 菜单显示与角色一致性

请按“测试步骤 + 预期结果”格式输出。
```

### 学生活动

完成 5 个验证动作：

1. 未登录访问 `/admin/panel`
2. `tom / USER` 登录访问 `/admin/panel`
3. `admin / ADMIN` 登录访问 `/admin/panel`
4. 登出后再次访问后台
5. 检查菜单是否正确显示

### 教师追问

“如果前端菜单不显示后台入口，但地址栏输入 `/admin/panel` 仍能打开，这算安全吗？”

### 学生回答

预期：

“不安全，服务端必须拦截。”

## 39-42分钟：展示修复结果，形成闭环

### 教师讲什么

“我们回到一开始的问题。现在请大家看，系统是否已经完成了从‘能登录’到‘能守住后台’的转变。”

### 教师操作

重新演示三条路径：

1. 未登录 -> 拒绝或重定向
2. 普通用户 -> 403 或拒绝
3. 管理员 -> 正常访问

### 教师总结语

“今天 AI 帮助我们更快地发现问题、定位知识点、生成验证路径；但真正让系统变得安全的，是大家对认证、授权、Session、RBAC 和 Filter 的理解，以及你们的修改和验证。”

## 42-45分钟：课堂总结与课后延伸

### 教师讲什么

“今天这节课我希望大家记住三句话：

1. 认证回答‘你是谁’
2. 授权回答‘你能做什么’
3. AI 可以帮你发现问题，但不能替你理解问题”

“课后请大家继续完成两个扩展任务：

1. 将密码散列由示例方式升级为更安全的 `BCrypt`
2. 增加 `MANAGER` 角色，扩展权限控制”

### 教师收束语

“希望大家以后做系统，不只追求‘能运行’，更要追求‘可验证、可控制、可信赖’。”

## 六、课堂中的 AI 应用说明词

这些话可以在录像中自然说出。

### 说明词 1

“今天我们使用 AI 助教，不是为了让它代替我们写代码，而是为了让它帮助我们更快地发现问题、对照知识点并生成验证路径。”

### 说明词 2

“AI 的建议只是起点，不是结论。最后是否成立，必须由课程知识和系统验证来决定。”

### 说明词 3

“真正重要的不是 AI 给了什么答案，而是同学们能否说明：这个问题为什么属于认证、为什么属于授权、应该如何验证是否修复成功。”

## 七、建议的投影页面顺序

建议课堂 PPT 或投影依次展示：

1. 问题页面：普通用户误入后台
2. 本节目标：Authentication / Authorization / Session / RBAC / Filter
3. 分组任务页
4. AI 提示词页
5. 小组汇报页
6. 知识点对齐页
7. 最小测试清单页
8. 结果验证页
9. 课堂总结页

## 八、建议的板书内容

可板书：

```text
Authentication -> 你是谁
Authorization  -> 你能做什么
Session        -> 保存登录状态
Filter         -> 统一控制入口
RBAC           -> 按角色授权
```

## 九、拍摄提醒

课堂实录中务必拍到：

1. 学生使用 AI 工具的画面
2. AI 输出问题诊断结果
3. 学生讨论 AI 结果的画面
4. 教师基于 AI 输出进行纠偏
5. 修改前后访问结果对比

如果只拍教师讲授而不拍学生使用 AI，人工智能赛道的说服力会明显下降。

## 十、备用方案

如果课堂现场 AI 网络不稳定，可准备：

1. 事先保存好的 AI 诊断结果截图
2. 事先生成的测试清单
3. 本地版任务单

但即使如此，也应尽量保留至少一段真实 AI 交互过程。

## 十一、一句话录课执行版

## `用“普通用户误入后台”的真实案例引发问题，让学生分组借助 AI 完成认证、授权与视图一致性的代码诊断、知识点核查、关键修复与行为验证，最终形成“问题发现—修改实现—结果验证”的完整课堂闭环。`
