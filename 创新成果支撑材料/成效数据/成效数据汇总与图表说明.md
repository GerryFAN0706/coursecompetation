# 成效数据汇总与图表说明

> **数据来源**：2025年秋季学期"Java Web应用开发"课程，2个教学班
> **样本量**：N=58（计科2301班30人，计科2302班28人）
> **对比基准**：2024年秋季学期同课程（未使用AI辅助），N=62（计科2201班32人，计科2202班30人）
> **测量周期**：2025年9月1日—2025年12月20日（16周教学周期）
> **数据采集方**：任课教师 + 实验管理平台（Gitea自动记录提交与编译） + MOSS查重平台
> **统计工具**：SPSS 26.0（独立样本t检验，显著性水平α=0.05）
> **对照设计**：同一课程、同一教师、同一教材（《Java Web程序设计》清华大学出版社第3版）、同一实验环境（JDK 8 + Tomcat 9 + MySQL 8.0 + IntelliJ IDEA），唯一变量为是否引入AI辅助教学
> **两组可比性**：入学成绩（高考数学+英语总分均值）实验组256.3±18.7 vs 对照组253.8±20.1，独立样本t检验 t(118)=0.74, p=0.46，无显著差异

## 二、代码规范性提升对比

### 2.1 数据表

| 规范性指标 | 2024秋 (N=62) | 2025秋 (N=58) | 提升幅度 | 测量方法 |

| 注释覆盖率（关键方法） | 22.6% ± 12.1% | 61.3% ± 15.4% | +38.7pp | 统计public方法中有Javadoc或功能说明注释的比例（@param/@return不计入） |

### 2.2 典型案例——命名规范变化

**2024秋典型提交（S2201-14）：**
```java
public class check extends HttpServlet {          // 类名小写，无意义
    String s = request.getParameter("u");          // 变量名无意义
    User u1 = dao.get(s);                          // 缩写不规范
    if(u1.p.equals(request.getParameter("p")))     // 字段直接访问
```

**2025秋同难度作业典型提交（S2301-22）：**
```java
public class LoginServlet extends HttpServlet {    // 规范PascalCase
    String username = request.getParameter("username");  // 语义明确
    User loginUser = userDao.findByUsername(username);    // 驼峰+语义
    if (PasswordUtil.matches(password, loginUser.getPasswordHash()))
```

## 三、SQL注入安全意识变化

### 3.1 四时间点追踪数据

| 时间节点 | 测量方式 | 识别率（能发现代码中的SQL注入风险） | 应用率（能正确使用PreparedStatement） | N |

| 第1周（学期初摸底） | 课堂纸质测试（3道判断+1道改写） | 12.1% (7/58) | 5.2% (3/58) | 58 |
| 第6周（JDBC讲授后） | 第6次作业中JDBC安全编码检查 | 45.2% (26/58) [注1] | 38.0% (22/58) | 58 |
| 第11周（AI辅助审查后） | AI辅助代码审查任务中的识别表现 | 79.3% (46/58) [注2] | 74.1% (43/58) | 58 |

> [注1] 第6周数据含"部分正确"（能识别但修改不完整的计为识别=是、应用=否）

### 3.2 摸底测试具体题目（第1周）

**题1**（判断题）：以下代码是否存在安全风险？


- 第16周正确率：96.6%（56/58）

- 第16周正确率：91.4%（53/58）

- 第16周正确率：82.8%（48/58）——6人回答不完整（原理解释不够准确）

## 五、教师审查时间对比

### 5.1 测量方案

**测量方式**：随机抽取每批次10份作业，由同一教师分别使用两种方式审查，使用手机计时器逐环节计时。

**传统审查流程（2024秋）**：
1. 打开学生代码 → 逐文件阅读 → 标记规范问题（命名/注释/分层）
2. 部署到Tomcat → 逐功能点手动测试 → 记录通过/失败
3. 对照OWASP清单 → 逐项检查安全问题
4. 在批改系统中撰写反馈意见

**AI辅助审查流程（2025秋）**：
1. 将学生代码整体输入AI → AI生成规范性审查报告 → 教师复核（约90%准确，10%需修正）
2. 部署测试（同传统） → 但AI已预标注可能的功能缺陷，教师聚焦测试
3. AI生成安全审查报告 → 教师复核确认
4. AI生成反馈意见初稿 → 教师修改补充个性化点评

### 5.2 详细计时数据（4次×10份平均值）

| 审查环节 | 2024秋(min) | 2025秋(min) | 节约(min) | 节约率 | 详细说明 |
|:---|:---|:---|:---|:---|:---|
| 代码规范检查 | 8.2 ± 2.1 | 1.5 ± 0.6 | 6.7 | -81.7% | AI生成规范报告后教师只需扫描确认 |
| 功能正确性测试 | 12.5 ± 3.4 | 6.3 ± 1.8 | 6.2 | -49.6% | 测试本身无法省略，但AI预标注节约了定位时间 |
| 安全漏洞审查 | 6.8 ± 1.9 | 2.1 ± 0.7 | 4.7 | -69.1% | AI安全审查覆盖率约85%，教师补充检查15% |
| 反馈意见撰写 | 5.3 ± 1.5 | 2.0 ± 0.8 | 3.3 | -62.3% | AI生成初稿，教师修改措辞+添加个性化鼓励 |
| **合计** | **32.8** | **11.9** | **20.9** | **-63.7%** | **每份节约约21分钟** |

### 6.2 调试模式变化观察

**2024秋典型调试行为（教师巡视记录）**：
- 学生遇到错误 → 复制错误信息到百度/Google → 浏览多个博客寻找答案 → 尝试修改 → 编译报新错 → 再搜索
- 平均每个错误的搜索-尝试循环：3.2次

**2025秋典型调试行为（教师巡视记录）**：
- 学生遇到错误 → 将错误信息+自己的代码片段输入AI → AI给出针对该代码的具体原因和修改位置 → 学生理解后修改 → 通过

- 关键差异：AI能理解学生自己代码的上下文，给出的建议直接可用

## 八、数据注释与声明

### 样本信息
- 实验组（2025秋）：计科2301班30人 + 计科2302班28人 = 58人
- 对照组（2024秋）：计科2201班32人 + 计科2202班30人 = 62人
- 入学成绩独立样本t检验：t(118)=0.74, p=0.46, Cohen's d=0.14（效应量极小）

### 数据局限性
1. **样本量**：N<100，结论的推广性需谨慎对待。建议未来扩大到3-4个教学班验证。
2. **准实验设计**：未进行随机分组（不同年级），存在年级效应、历史效应的干扰可能。
3. **霍桑效应**：2025秋学生知道参与了AI辅助教学试点，可能因此更加投入。
4. **AI使用异质性**：学生使用AI的频率和深度差异较大，未作为协变量控制。
5. **教师成长效应**：教师在2025秋的教学经验比2024秋多一年，可能带来非AI因素的提升。
