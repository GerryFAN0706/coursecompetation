## 二、各创新模块详细设计

### 创新一：课程专属教学智能体「WebDev助教」

#### 定位
不是通用ChatGPT，而是**只懂这门课**的AI助教——基于课程大纲、16讲课件、实验手册、常见错误库构建的RAG系统。

#### 功能设计

| 功能 | 描述 | 应用阶段 |
|:---|:---|:---|
| 课程问答 | 学生提问→从课程知识库检索→结合上下文回答 | 课前/课后 |
| 预习引导 | 每讲课前推送3个引导问题，学生作答后AI点评 | 课前 |
| 错误诊断 | 学生粘贴报错信息→AI结合课程内容定位问题层（Controller/Service/DAO） | 课中/课后 |
| 知识点检查 | 学生提交代码→AI对照课程大纲检查知识点覆盖情况 | 课后 |
| 学情热力图 | 教师看板：全班提问频次、知识薄弱点、高频错误类型 | 教师端 |

#### 技术实现方案

```
技术栈：
├── 前端：HTML + CSS + JavaScript（简易聊天界面）
│   └── 可作为课程的"活教材"——学生在学Java Web的同时使用Java Web产品
├── 后端：Python Flask / FastAPI（快速开发）
│   ├── 大模型API：DeepSeek / 通义千问（国产大模型，加分项）
│   ├── 向量数据库：ChromaDB / FAISS（存储课程文档向量）
│   └── RAG Pipeline：课件+教案+实验手册→分段→embedding→检索→生成
├── 数据：SQLite存储对话记录
│   └── 教师看板：Echarts可视化学情分析
└── 部署：校内服务器 / 阿里云学生机
```

#### 竞赛亮点
- **官方文件明确要求"教学智能体"**——这是最直接的回应
- 使用**国产大模型**（DeepSeek/通义千问），体现自主可控
- 24小时可用，真正改变学生的学习方式
- 自然产生学情数据，支撑"数据驱动"叙事

#### 在课堂实录中的呈现
- 课堂开始时，教师展示"课前预习数据"：全班58人中42人向智能体提问，热点问题是"Session和Cookie的区别"
- 据此调整课堂重点，体现**数据驱动教学决策**

### 创新二：AI驱动的安全攻防闯关「WebSec挑战」

#### 定位
把安全教学从"老师讲概念"变成"学生玩闯关"——AI动态生成有安全漏洞的Java Web代码，学生找漏洞、修代码、过关。

#### 关卡设计（5关，难度递增）

| 关卡 | 漏洞类型 | 知识点 | AI角色 | 时间 |
|:---|:---|:---|:---|:---|
| 第1关 | SQL注入 | PreparedStatement vs 字符串拼接 | AI生成含漏洞的DAO代码 | 3min |
| 第2关 | 越权访问 | Filter权限检查缺失 | AI生成缺少权限校验的Servlet | 4min |
| 第3关 | Session劫持 | Session固定攻击、invalidate | AI生成不安全的登录逻辑 | 4min |
| 第4关 | XSS跨站脚本 | 输出转义、JSTL `<c:out>` | AI生成未转义的JSP页面 | 4min |
| 第5关（BOSS关） | 综合漏洞 | 上述所有 + CSRF | AI综合生成完整模块 | 5min |

#### 核心机制

```
学生操作流程：
1. 系统展示一段有漏洞的Java Web代码（AI为每人生成不同版本）
2. 学生找出漏洞位置，写出修复方案
3. 提交修复代码 → 后端自动运行测试用例 → 判定通过/失败
4. 通过 → 进入下一关，AI给出安全知识点总结
5. 失败 → AI给出提示（不是答案），学生重试

教师看板（实时）：
- 全班进度大屏：谁在第几关、通过率、平均用时
- 可即时发现共性薄弱点，进行课堂干预
```

#### 技术实现方案

```
技术栈：
├── 前端：简单Web页面（代码编辑器 + 关卡进度条 + 排行榜）
│   └── 使用 CodeMirror 做代码编辑器
├── 后端：
│   ├── 题目生成：调用大模型API，用prompt模板生成带漏洞的代码
│   │   └── prompt模板：「生成一个Java Servlet处理用户登录的代码，
│   │       其中故意包含{漏洞类型}漏洞，场景为{业务场景}」
│   ├── 判定引擎：JUnit测试用例自动验证修复结果
│   │   └── 每关预设5-8个测试用例（含正常流程+攻击流程）
│   └── 进度追踪：记录每人每关的尝试次数、用时、修复方案
└── 数据：生成成效证据（通过率趋势、漏洞识别率提升等）
```

#### 竞赛亮点
- **游戏化教学**——学生参与度极高，视频效果好
- **AI动态出题**——每人不同题目，天然防抄袭
- **实时数据大屏**——教师可做课堂干预，体现数据驱动
- 产出丰富的过程数据（通过率、用时、薄弱关卡）

#### 在课堂实录中的呈现
- 课中10分钟：学生完成2-3个关卡
- 教师看大屏发现"第2关通过率仅45%"→ 暂停闯关 → 集中讲解越权访问
- 体现"师生机协同"+ "数据驱动即时干预"

### 创新三：AI辅助团队协作项目教练「TeamCoach」

#### 定位
团队项目中的AI"项目经理"——自动分析每个成员的Git贡献，生成团队健康报告，防止搭便车，促进真正的工程化协作。

#### 功能设计

| 功能 | 描述 | 数据来源 |
|:---|:---|:---|
| 贡献度分析 | 每人代码行数、提交频率、修改文件分布 | Git log |
| 代码质量评分 | 代码规范性、注释率、重复度、圈复杂度 | 静态分析 + AI |
| 分工合理性 | 检测是否有人只改JSP/只改配置文件 | Git diff 分类 |
| 搭便车预警 | 连续N天无提交、提交集中在截止日前1天 | 提交时间分析 |
| 周报生成 | AI每周为每个团队生成项目健康报告 | 综合以上数据 |
| 代码风格一致性 | 命名规范、缩进、包结构是否统一 | AST分析 + AI |

#### AI周报样例

```
━━━ TeamCoach 周报 · 第3组 · 第12周 ━━━

📊 本周概况
  总提交：47次 | 活跃成员：4/4 | 代码新增：+823行

👥 成员贡献
  张三：18次提交，主要修改 Controller层（67%）  ★ 核心开发者
  李四：12次提交，主要修改 DAO层（54%）         ✓ 稳定贡献
  王五：14次提交，主要修改 JSP页面（71%）       ✓ 前端专注
  赵六：3次提交，均在周日，仅修改README          ⚠ 贡献不足

⚠ 风险提示
  1. 赵六本周贡献显著低于团队平均，建议教师关注
  2. Service层无人负责，可能成为集成瓶颈
  3. 测试代码覆盖率为0%，建议补充单元测试

🎯 下周建议
  1. 赵六可接手Service层开发，平衡分工
  2. 全组进行一次代码走查，统一命名规范
  3. 截止日前3天设置里程碑检查点
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

#### 技术实现方案

```
技术栈：
├── 数据采集：
│   ├── Git log 解析（Python脚本，按作者/日期/文件分类统计）
│   ├── 代码静态分析（Checkstyle / PMD / 自定义规则）
│   └── 相似度检测（MOSS或自建工具）
├── AI分析：
│   ├── 将统计数据 + 代码片段喂给大模型
│   └── prompt：「作为项目教练，分析以下团队的Git贡献数据，
│       生成周报，指出风险并给出建议」
├── 输出：
│   ├── Markdown周报（自动生成）
│   ├── 教师看板：全班各组对比、预警列表
│   └── 学生端：个人贡献雷达图
└── 部署：定时任务（每周日自动运行，周一早上推送）
```

#### 竞赛亮点
- 解决团队项目中"搭便车"的**真实教学痛点**
- 用**数据驱动**而非主观判断评价团队贡献
- 体现工程化思维培养（Git规范、持续集成理念）
- 过程性评价，不只看最终成果

#### 在课堂实录中的呈现
- 课堂开始时，教师展示"上周TeamCoach报告"：
  - "第3组的赵六同学连续两周贡献不足，今天课堂上我会重点关注"
  - "全班Service层代码质量偏低，今天补充讲解设计模式"
- 体现**数据驱动教学干预**

### 创新四：AI代码评审对战「Code Review Battle」

#### 定位
把代码评审变成"人 vs AI"的对抗性学习——学生和AI同时审查一段代码，比谁找的问题更准、更全、更有价值。

#### 对战机制

```
对战流程（15分钟）：

第1轮：人工审查（5min）
  ├── 教师投影一段有缺陷的Java Web代码
  ├── 学生分组讨论，找出所有问题
  └── 每组提交审查报告（问题列表 + 修复建议）

第2轮：AI审查（2min）
  ├── 教师将同一段代码提交给AI
  └── AI输出审查报告

第3轮：对比与辩论（8min）
  ├── 对比人工 vs AI的发现
  │   ├── AI找到了但学生没找到的 → 学习机会
  │   ├── 学生找到了但AI没找到的 → 人类优势
  │   └── AI给出了错误建议的 → 批判性思维训练
  ├── 小组辩论：哪些AI建议该采纳？哪些该拒绝？为什么？
  └── 教师总结：AI在安全审查中的能力边界
```

#### 评分维度

| 维度 | 人工 | AI | 教学意义 |
|:---|:---|:---|:---|
| 发现问题数量 | 通常较少 | 通常较多 | AI覆盖面广 |
| 问题优先级判断 | 通常更准 | 可能误判 | 人的业务理解更深 |
| 修复方案质量 | 贴合项目 | 可能过度设计 | 人更了解上下文 |
| 安全风险评估 | 需要经验 | 模式匹配强 | 互补合作最优 |

#### 竞赛亮点
- 直接回应"学生会不会过度依赖AI"的社会关切
- 培养**批判性思维**——不盲从AI
- "人机协同"的最佳诠释：不是人用AI，而是人和AI各有优势
- 课堂效果好：对抗性活动天然有张力，视频表现力强

#### 在课堂实录中的呈现
- 课中15分钟的核心环节
- 学生发现AI遗漏了一个越权访问漏洞 → 全班鼓掌 → 教师总结"AI不是万能的"
- 教师展示AI误报的一个"问题" → 引导学生分析为什么这不是真正的问题

## 三、课堂教学实录选题分析

### 为什么选第十五讲「用户认证与权限控制」

这是唯一一讲能**在45分钟内自然展现全部五维融合创新**的课程内容：

| 对比维度 | 第十讲 MVC综合实战 | 第十五讲 用户认证与权限控制 | 第六讲 JDBC |
|:---|:---|:---|:---|
| 安全攻防闯关 | ✗ 无安全内容 | ★ SQL注入+越权+Session | △ 仅SQL注入 |
| AI代码评审对战 | △ 架构问题 | ★ 安全漏洞（隐蔽性强，对战效果好） | △ SQL问题 |
| 教学智能体展示 | ○ 可展示 | ★ 展示预习数据+课堂即时查询 | ○ 可展示 |
| 团队项目教练展示 | △ 可展示报告 | ★ 展示周报+据此调整课堂 | ✗ 无团队内容 |
| 知识综合性 | 中（MVC三层） | **极高**（Session+Filter+DAO+MVC+安全） | 低（仅JDBC） |
| 学生参与度 | 中 | **极高**（攻防+对战+分组） | 中 |
| 视频表现力 | 中 | **极高**（有对抗、有惊喜、有数据大屏） | 低 |
| 思政融合 | 工匠精神 | **依法治网+权责明晰+安全合规** | 数据安全 |

**结论：第十五讲是最佳且唯一的选择。**
