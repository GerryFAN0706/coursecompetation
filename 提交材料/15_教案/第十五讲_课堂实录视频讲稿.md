### [15:00-18:00] Filter权限检查 · 核心讲解

**画面**：课件——"Filter权限检查 · 核心讲解"页面，展示RoleBasedAuthFilter完整代码

**教师动作**：站在讲台前，边指代码边讲解，语速适中

**教师台词**：
> 好，大家看这段代码。这是解决越权访问问题的核心——RoleBasedAuthFilter。
>
> 我们刚才看到的那个漏洞代码里，Servlet只检查了一件事：`session != null` 并且 `session.getAttribute("user") != null`。翻译成人话就是——"你登录了吗？"如果登录了，就放你进去。
>
> 但问题在于，Tom也登录了！他确实是一个合法用户，只是他的角色不是ADMIN。所以我们需要的不是检查"是否登录"，而是检查"角色是否匹配"。
>
> 看这段修复代码：
>
> ```java
> User user = (User) session.getAttribute("user");
> if (user != null && "ADMIN".equals(user.getRole())) {
>     chain.doFilter(req, res);   // ADMIN放行
> } else if (user != null) {
>     response.sendError(403);    // 已登录但非ADMIN → 403 Forbidden
> } else {
>     response.sendRedirect("/login.jsp");  // 未登录 → 重定向
> }
> ```
>
> 三种情况，三种处理方式：管理员放行、普通用户返回403禁止访问、未登录跳转到登录页。
>
> 这里有一个关键点——为什么用 `"ADMIN".equals(user.getRole())` 而不是 `user.getRole().equals("ADMIN")`？
>
> （等一两秒）因为如果 `getRole()` 返回null，后者会抛空指针异常，而前者不会。这是一个防御性编程的好习惯，defensive programming。
>
> 还有一个重点：这个Filter用 `@WebFilter("/admin/*")` 注解，意思是它只拦截 `/admin/` 路径下的所有请求。这就是Filter在权限控制中的核心作用——统一入口、统一控制。
>
> Authentication回答"你是谁"——这是LoginServlet的工作；Authorization回答"你能做什么"——这是Filter的工作。先认证，后授权，顺序不能反。

**学生活动**：认真听讲，对照自己闯关中遇到的困难理解代码逻辑

**AI工具操作**：无（纯教师讲解）

**拍摄提示**：教师与代码投影同框，确保代码字号足够大、清晰可读。教师指到哪行代码，镜头可适当推近。讲到"三种情况三种处理"时，可用激光笔或手指依次指出。

### [36:00-39:00] 教师带领修复三处关键代码

**画面**：IDE界面——分别打开三个文件进行修复

**教师动作**：在教师机上操作代码修改，边改边讲解

**教师台词**：

> **修复1：RoleBasedAuthFilter**
>
> 首先，打开 `RoleBasedAuthFilter.java`。现在的代码只检查了 `isLoggedIn`——"是否登录"。我们要改成检查 `user.getRole()`——"角色是否为ADMIN"。
>
> （边改边说）把这一行改掉……`User user = (User) session.getAttribute("user");`……然后判断 `"ADMIN".equals(user.getRole())`……ADMIN放行，非ADMIN返回403，未登录重定向到登录页。
>
> 这就是从Authentication-only升级到Authorization的关键一步。
>
> **修复2：index.jsp**
>
> 第二个文件，`index.jsp`。现在的菜单代码是——只要登录了就显示"管理后台"入口。我们要改成按角色条件显示。
>
> （边改边说）加上 `<c:if test="${sessionScope.user.role == 'ADMIN'}">`……只有当用户角色是ADMIN时，才渲染这个管理后台的链接。
>
> 但要记住——前端隐藏菜单只是用户体验优化，真正的安全边界在后端Filter。这两层防护必须同时存在，缺一不可。
>
> **修复3：LogoutServlet**
>
> 第三个文件，`LogoutServlet.java`。把 `session.removeAttribute("user")` 改成 `session.invalidate()`。
>
> （边改边说）`removeAttribute` 只删除了一个属性，Session对象本身还活着，里面可能还有其他属性残留。而 `invalidate()` 是彻底销毁整个Session，所有属性一并清除，服务器端也不再维持这个会话。
>
> 一个是"擦掉名牌"，一个是"销毁通行证"。效果完全不同。

## 附录F：教师备忘

1. **语速控制**：每分钟100-110字，留出停顿和互动时间。不要赶进度。
2. **自然过渡**：各阶段之间不要有"下面进入第X环节"这种生硬过渡，用问题或数据自然衔接。
3. **眼神交流**：不要一直看屏幕，经常看向学生。提问时扫视全班。
4. **AI工具操作**：每次打开AI平台时，稍作停顿让镜头捕捉到工具界面。
5. **学生互动**：不要所有问答都自问自答，至少让3-5位学生发言。
6. **数据驱动**：每次展示数据后都要说明"这个数据告诉我什么/我据此做了什么决策"，体现数据驱动教学。
7. **双语要求**：关键术语第一次出现时给出中英文对照，后续可只说英文或只说中文。核心术语包括：Authentication、Authorization、Session、RBAC、Filter、Invalidate、Cache-Control。
8. **课程思政**：思政部分要自然融入总结，不要单独拎出来说"下面是思政环节"，这样会很生硬。
