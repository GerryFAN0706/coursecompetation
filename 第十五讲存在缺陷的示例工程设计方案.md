# 第十五讲存在缺陷的示例工程设计方案

> 课程：`Java Web应用开发`  
> 对应课堂：`第十五讲 用户认证与权限控制`  
> 用途：用于人工智能赛道课堂实录，作为“待诊断、待修复、待验证”的课堂问题工程

## 一、设计目标

这份示例工程不是为了做一个“完整正确”的系统，而是为了做一个：

## `问题真实、结构清晰、便于课堂诊断、适合AI介入、修复后效果明显`

的课堂案例工程。

理想状态下，这个工程应该满足：

1. 学生一看就能理解业务场景。
2. 缺陷不是“语法报错”，而是“能运行但不安全、不规范”。
3. 问题分布在 `认证 / 授权 / 视图一致性` 三个层面。
4. AI 能帮助发现问题，但不会一眼直接给出唯一答案。
5. 修复后可以通过 403 / 重定向 / 菜单变化直观看到结果。

## 二、推荐项目主题

建议使用一个轻量、熟悉、可快速理解的项目：

## `课程作业管理系统`

也可以用以下替代主题：

1. 图书管理系统
2. 实验室预约系统
3. 在线选课系统

但从“角色权限”直观性看，`课程作业管理系统` 最合适，因为它天然有：

1. 学生用户 `USER`
2. 教师/管理员用户 `ADMIN`
3. 普通页面
4. 后台管理页面

## 三、推荐工程最小功能范围

课堂工程只保留最小必要功能，不要做太复杂。

建议包含：

1. `login.jsp`
   - 登录页
2. `index.jsp`
   - 首页
3. `admin/panel.jsp`
   - 后台管理页
4. `LoginServlet`
5. `LogoutServlet`
6. `RoleBasedAuthFilter`
7. `UserService`
8. `UserDao`
9. `User` 实体类

## 四、推荐角色与账号

建议预置两个账号即可：

1. `admin`
   - 角色：`ADMIN`
   - 功能：可进入后台
2. `tom`
   - 角色：`USER`
   - 功能：不能进入后台

如果你想增强课堂层次，也可以增加：

3. `manager`
   - 角色：`MANAGER`
   - 用于课后扩展，不建议录课主阶段加入

## 五、推荐页面与访问路径

建议路径设计如下：

| 页面/资源 | 路径 | 说明 |
| :--- | :--- | :--- |
| 登录页 | `/login.jsp` | 未登录可访问 |
| 登录提交 | `/login` | `LoginServlet` |
| 首页 | `/index.jsp` | 所有人可访问 |
| 后台页 | `/admin/panel.jsp` | 仅 `ADMIN` 可访问 |
| 退出登录 | `/logout` | `LogoutServlet` |

## 六、最适合故意埋入的缺陷

核心原则：

## `埋“逻辑缺陷”，不埋“编译错误”`

课堂里最怕学生花时间修语法错误、路径错误、环境错误。  
最适合录课的，是“系统能运行，但行为不正确”的问题。

下面给出推荐的 `5 个候选缺陷`，其中建议主录课使用前 `3 个`，第 `4-5 个` 用于扩展或课后。

---

## 缺陷 1：只判断已登录，不判断角色

### 推荐指数

`★★★★★`

### 为什么最适合录课

这是整节课最核心、最典型、最容易看出修改效果的问题。

### 缺陷表现

`RoleBasedAuthFilter` 只判断：

- Session 中有没有用户对象

但不判断：

- 用户角色是不是 `ADMIN`

### 错误行为

1. `tom / USER` 登录后
2. 直接访问 `/admin/panel.jsp`
3. 系统居然允许访问

### 适合对应的小组

`B组：授权组`

### 适合对应的知识点

1. Authorization
2. RBAC
3. Filter

### 建议保留的错误代码风格

```java
HttpSession session = request.getSession(false);
User u = (session == null) ? null : (User) session.getAttribute("user");
if (u != null) {
    chain.doFilter(req, res);
} else {
    response.sendRedirect(request.getContextPath() + "/login.jsp");
}
```

### 正确修复方向

应改为：

1. 先判断是否登录
2. 再判断是否为 `ADMIN`
3. 非 ADMIN 则返回 403 或跳转提示页

---

## 缺陷 2：JSP 菜单只按“是否登录”显示，不按角色显示

### 推荐指数

`★★★★★`

### 为什么适合录课

它能很好地引出一个课堂亮点：

## `前端隐藏入口 != 真正安全`

### 缺陷表现

首页 `index.jsp` 中只要用户已登录，就显示“管理后台”入口。

### 错误行为

1. `tom / USER` 登录后
2. 首页居然也能看到“管理后台”按钮

### 适合对应的小组

`C组：视图一致性组`

### 适合对应的知识点

1. Session
2. Authorization
3. View Rendering

### 建议保留的错误代码风格

```jsp
<c:if test="${sessionScope.user != null}">
    <a href="${pageContext.request.contextPath}/admin/panel.jsp">管理后台</a>
</c:if>
```

### 正确修复方向

应改为：

1. 只有 `sessionScope.user.role == 'ADMIN'` 才显示后台入口
2. 同时强调：即使前端不显示入口，后端仍必须拦截

---

## 缺陷 3：登出逻辑不彻底，只删属性不失效会话

### 推荐指数

`★★★★☆`

### 为什么适合录课

它不如“越权访问”那么显眼，但非常适合体现：

1. Session 生命周期理解
2. “退出登录”不是简单返回登录页
3. AI 可协助识别会话残留问题

### 缺陷表现

`LogoutServlet` 中只做：

```java
session.removeAttribute("user");
response.sendRedirect("login.jsp");
```

而没有：

```java
session.invalidate();
```

### 错误行为

可能出现：

1. 会话残留
2. 重新登录后状态异常
3. 某些页面刷新逻辑混乱

### 适合对应的小组

`A组：认证组`

### 适合对应的知识点

1. Authentication
2. Session-Based Login
3. Logout

### 正确修复方向

1. 使用 `getSession(false)`
2. 非空时调用 `invalidate()`
3. 再跳转到登录页

---

## 缺陷 4：登录成功后直接保存完整用户对象，但未做最小必要会话设计

### 推荐指数

`★★★☆☆`

### 为什么适合作为拓展，不建议做主问题

这个问题更偏工程优化和安全意识，不如前三个直观。

### 缺陷表现

登录后直接把包含过多信息的 `User` 对象完整塞入 Session，例如：

1. 敏感字段
2. 不必要字段
3. 调试信息

### 课堂价值

可用于教师拔高：

- Session 中应尽量保存最小必要信息

### 建议

不作为主录课问题，可作为教师补充点评点。

---

## 缺陷 5：密码仍使用简单散列示例且无课堂边界说明

### 推荐指数

`★★★☆☆`

### 为什么适合作为课后拓展

这更适合在课后或汇报中作为“进一步优化”，而不适合作为主课堂修复点。

### 缺陷表现

密码工具类使用简单 SHA-256 示例实现。

### 课堂价值

可用于补充：

1. 示例只为讲原理
2. 真实项目更推荐 `BCrypt/Argon2`

### 建议

不要在录课主阶段让学生深入改密码算法，否则容易偏离课堂主线。

## 七、最推荐的录课缺陷组合

如果你只想录一节最稳的课堂，建议只埋下面 `3 个问题`：

## `推荐组合：1 + 2 + 3`

即：

1. `过滤器只判断登录，不判断角色`
2. `JSP 菜单显示与角色不一致`
3. `登出不彻底，会话失效处理不规范`

这是最优组合，因为它们分别覆盖：

1. 授权控制
2. 视图一致性
3. 认证与会话管理

而且刚好能分给 3 个学习小组。

## 八、不建议埋的缺陷类型

下面这些问题不适合课堂录课主线。

## 1. 编译错误

例如：

1. 少分号
2. import 错误
3. 包名错误

原因：

- 太低级
- 浪费课堂时间
- AI 一眼就修掉，教学价值低

## 2. 环境配置错误

例如：

1. 数据库连不上
2. Tomcat 路径错误
3. 驱动缺失

原因：

- 不利于课堂节奏
- 修复依赖环境，不利于录课稳定

## 3. 路径拼写错误

例如：

1. JSP 文件路径错
2. redirect 地址写错

原因：

- 只是调试，不是核心知识

## 4. 过深的密码学问题

例如：

1. 盐值设计
2. 彩虹表对抗细节
3. 会话固定攻击完整实现

原因：

- 太深，容易把一节 Java Web 课拍成安全专题课

## 十、推荐故意埋错的位置

为了方便课堂分组与 AI 审查，建议把错误有意识地分散在 3 个文件中。

## 1. `RoleBasedAuthFilter.java`

埋错内容：

1. 只判断 `user != null`
2. 不判断 `role`

课堂价值：

- 最核心的授权问题

## 2. `index.jsp`

埋错内容：

1. 登录后统一显示“管理后台”
2. 不区分角色

课堂价值：

- 视图与权限一致性问题

## 3. `LogoutServlet.java`

埋错内容：

1. 只移除属性，不销毁 Session

课堂价值：

- 会话失效机制问题

## 十一、推荐课堂前的“初始错误行为演示”

上课前，教师最好先确认这 3 个错误行为能稳定复现。

## 演示行为 1

`tom / USER` 登录后，访问 `/admin/panel.jsp`

预期错误表现：

- 居然成功进入后台

## 演示行为 2

`tom / USER` 登录后打开首页

预期错误表现：

- 首页显示“管理后台”入口

## 演示行为 3

登录后退出，再做刷新或重新访问受保护资源

预期错误表现：

- 某些情况下状态处理不够彻底或逻辑不清楚

## 十二、推荐课堂中的分组映射

| 小组 | 检查文件 | 主要问题 | 对应知识点 |
| :--- | :--- | :--- | :--- |
| A组 | `LoginServlet.java` `LogoutServlet.java` | 登录态建立与失效不规范 | Authentication / Session |
| B组 | `RoleBasedAuthFilter.java` | 只登录不鉴权 | Authorization / RBAC / Filter |
| C组 | `index.jsp` | 菜单显示与权限不一致 | Session / View / Authorization |

## 十三、推荐课堂中的 AI 使用点

这个问题工程最适合 AI 介入的地方如下：

## 1. 查找权限漏洞

AI 适合帮助学生判断：

1. 过滤器是否真正做了角色校验
2. 认证和授权是否被混淆

## 2. 对照课程知识点

AI 适合帮助学生把问题映射到：

1. Authentication
2. Authorization
3. Session-Based Login
4. RBAC
5. Filter

## 3. 生成最小验证清单

AI 适合帮助学生快速生成：

1. 未登录访问后台测试
2. 普通用户访问后台测试
3. 管理员访问后台测试
4. 注销后访问测试
5. 菜单显示一致性测试

## 十四、修复后的正确行为

这节课修复后，系统应该达到：

1. 未登录访问后台：
   - 被拦截
2. `tom / USER` 登录访问后台：
   - 被拦截或返回 403
3. `admin / ADMIN` 登录访问后台：
   - 正常访问
4. `tom / USER` 登录首页：
   - 不显示管理后台入口
5. `admin / ADMIN` 登录首页：
   - 显示管理后台入口
6. 登出后访问后台：
   - 再次被拦截

## 十五、建议你实际准备两份工程

为了录课和后续复盘更顺畅，建议准备两份版本：

## 1. `问题版工程`

特点：

1. 带有上述 3 个缺陷
2. 用于课堂诊断与修复

## 2. `修复版工程`

特点：

1. 已完成标准修复
2. 用于课后展示、备用演示、录课容错

如果课堂临场出现问题，也可以及时切换到修复版演示结果。

## 十六、录课时最理想的课堂节奏

建议按下面顺序组织：

1. 教师演示问题版行为
2. 学生分组用 AI 审查代码
3. 每组锁定 1 个核心问题
4. 学生修改关键代码
5. 按 3 条路径验证结果
6. 教师总结“从登录到权限控制”的完整逻辑

## 十七、一句话最终方案

## `建议将课堂问题工程设计为一个“课程作业管理系统”的最小权限案例，并故意埋入“只登录不鉴权、菜单显示不分角色、登出不彻底”这 3 个逻辑缺陷，作为第十五讲课堂中最适合 AI 诊断、学生修复、结果可视化验证的录课问题组合。`
