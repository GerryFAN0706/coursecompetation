缺陷一：RoleBasedAuthFilter 越权访问 高危

Filter 拦截 /admin/* 请求，但只检查是否登录，不检查角色。任何登录用户都能访问管理后台。

问题代码 RoleBasedAuthFilter.java
filter/RoleBasedAuthFilter.java — doFilter()
10public void doFilter(ServletRequest req,
11    ServletResponse res, FilterChain chain) {
12  HttpServletRequest request = (HttpServletRequest) req;
13  HttpSession session = request.getSession(false);
14
15  boolean loggedIn = session != null
16      && Boolean.TRUE.equals(
17          session.getAttribute("isLoggedIn"));
18
19  if (loggedIn) {
20      chain.doFilter(req, res); // 放行所有登录用户!
21  } else {
22      response.sendRedirect("login.jsp");
23  }
24}
修复代码 RoleBasedAuthFilter.java
filter/RoleBasedAuthFilter.java — doFilter()
10public void doFilter(ServletRequest req,
11    ServletResponse res, FilterChain chain) {
12  HttpServletRequest request = (HttpServletRequest) req;
13  HttpSession session = request.getSession(false);
14
15  User user = (session != null)
16      ? (User) session.getAttribute("user")
17      : null;
18
19  if (user != null && "ADMIN".equals(user.getRole())) {
20      chain.doFilter(req, res); // 仅ADMIN放行
21  } else if (user != null) {
22      response.sendError(403); // 已登录非ADMIN→403
23  } else {
24      response.sendRedirect("login.jsp"); // 未登录→登录页
25  }
26}

涉及知识点

  • Filter doFilter()：Servlet规范中的请求拦截机制，在请求到达Servlet前执行
  • @WebFilter("/admin/*")：URL模式匹配，拦截所有 /admin/ 下的请求
  • HttpSession getAttribute()：从Session中获取属性，返回Object需强制转换
  • 认证 vs 授权：认证（Authentication）= 你是谁；授权（Authorization）= 你能做什么
  • HttpServletResponse.SC_FORBIDDEN (403)：标准HTTP状态码，表示服务器理解请求但拒绝执行

缺陷二：index.jsp 菜单越权 中危

首页导航对所有登录用户显示"管理后台"链接，不按角色区分。即使 Filter 修复了，用户仍能看到不该看到的入口。

问题代码 index.jsp
webapp/index.jsp — 导航菜单部分
38<c:choose>
39  <c:when test="${empty sessionScope.user}">
40    <a href="login.jsp">登录</a>
41  </c:when>
42  <c:when test="${sessionScope.isLoggedIn}">
43    <a href="logout">退出</a>
44    <a href="admin/panel.jsp">管理后台</a>
45  </c:when> <!-- 所有登录用户都能看到! -->
46</c:choose>
修复代码 index.jsp
webapp/index.jsp — 导航菜单部分
38<c:choose>
39  <c:when test="${empty sessionScope.user}">
40    <a href="login.jsp">登录</a>
41  </c:when>
42  <c:otherwise>
43    <a href="logout">退出</a>
44    <c:if test="${sessionScope.user.role == 'ADMIN'}">
45      <a href="admin/panel.jsp">管理后台</a>
46    </c:if> <!-- 仅ADMIN可见 -->
47  </c:otherwise>
48</c:choose>

涉及知识点

  • JSTL <c:if>：条件标签，test属性为true时渲染标签体内容
  • EL表达式 ${sessionScope.user.role}：通过点号导航访问JavaBean属性（调用getRole()）
  • 视图层权限控制：前端菜单隐藏是"体验优化"，不是安全屏障——真正的安全靠后端Filter
  • 纵深防御：Filter（后端拦截）+ JSP（前端隐藏）双重保护，即使一层被绕过另一层仍有效

缺陷三：LogoutServlet 会话残留 中危

退出登录时只移除了 user 属性，Session 中的 isLoggedIn、role 等属性仍然存在。可能导致会话劫持。

问题代码 LogoutServlet.java
controller/LogoutServlet.java — doGet()
12protected void doGet(HttpServletRequest request,
13    HttpServletResponse response) {
14  HttpSession session = request.getSession(false);
15  if (session != null) {
16      session.removeAttribute("user");
17      // isLoggedIn = true  仍残留!
18      // role = "USER"       仍残留!
19      // displayName = "Tom" 仍残留!
20  }
21  response.sendRedirect("login.jsp");
22}
Session 中 4 个属性只清除了 1 个，
isLoggedIn=true 残留导致 Filter 可能仍判定为已登录。
修复代码 LogoutServlet.java
controller/LogoutServlet.java — doGet()
12protected void doGet(HttpServletRequest request,
13    HttpServletResponse response) {
14  HttpSession session = request.getSession(false);
15  if (session != null) {
16      session.invalidate();
17      // 整个 Session 销毁:
18      // - 所有属性清除
19      // - Session ID 失效
20      // - Cookie JSESSIONID 作废
21  }
22  response.sendRedirect("login.jsp");
23}
invalidate() 一行代码解决所有问题：
属性清除 + Session ID 失效 + Cookie 作废。

涉及知识点

  • session.invalidate()：销毁整个Session对象，所有属性被清除，Session ID失效
  • session.removeAttribute()：只移除一个指定属性，Session和其他属性继续存在
  • Session固定攻击（Session Fixation）：如果旧Session ID可复用，攻击者可劫持用户身份
  • Cookie JSESSIONID：invalidate 后服务器不再认识该ID，即使浏览器Cookie仍在也无法复用

额外修复：LoginServlet Session 冗余存储

问题版在Session中存了4个属性（user + isLoggedIn + role + displayName），修复版只存1个User对象——单一数据源原则。

问题代码 LoginServlet.java
controller/LoginServlet.java — doPost()
22User user = userService.login(username, password);
23if (user != null) {
24    HttpSession session = request.getSession();
25    session.setAttribute("user", user);
26    session.setAttribute("isLoggedIn", true);
27    session.setAttribute("role", user.getRole());
28    session.setAttribute("displayName",
29        user.getDisplayName());
30    response.sendRedirect("index.jsp");
31}
4个属性冗余存储，修改 User 后还需同步更新 Session 中的副本。
LogoutServlet 需要逐个 removeAttribute。
修复代码 LoginServlet.java
controller/LoginServlet.java — doPost()
22User user = userService.login(username, password);
23if (user != null) {
24    HttpSession session = request.getSession();
25    session.setAttribute("user", user);
26    // 单一数据源：角色通过 user.getRole() 获取
27    // 登录状态通过 user != null 判断
28    // 显示名通过 user.getDisplayName() 获取
29    response.sendRedirect("index.jsp");
30}
1个对象 = 1个数据源。
JSP 中用 ${sessionScope.user.role}、${sessionScope.user.displayName} 直接访问。

涉及知识点

  • 单一数据源原则：同一信息只存一处，避免不一致（如修改role后Session中的副本未同步）
  • JavaBean 规范：User 类的 getRole()、getDisplayName() 遵循 getter 命名规范，EL 可直接导航
  • EL 点号导航：${sessionScope.user.role} 自动调用 user.getRole()，无需手动类型转换
