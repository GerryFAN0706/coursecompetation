## 一、课前提示词

课前提示词用于教学准备阶段，由教师在备课环节使用AI完成学情数据分析、差异化任务生成和教学资源准备，确保课堂活动精准匹配学生能力层次。

### Prompt 1：学情分析与学生分层

<div class="case-badges">
  <span>学情分析</span>
  <span>数据驱动</span>
  <span>A/B/C分层</span>
</div>

**使用场景：** 在课前根据学生历史作业成绩、实验完成度和课堂互动数据，将全班学生划分为A（挑战组）、B（进阶组）、C（基础组）三个层次，为后续差异化任务生成提供依据。

```prompt
你是一位Java Web课程的教学分析助手。请根据以下学生数据完成学情分析与分层。

## 输入数据
- 课程名称：Java Web应用开发
- 当前教学进度：[第N讲主题，例如：第十五讲 用户认证与权限控制]
- 学生数据（JSON或表格格式）：[粘贴学生成绩数据]
- 数据维度包括：前序实验得分、课堂测验正确率、代码规范评分、历史提交及时率

## 分析要求
1. 按综合表现将学生分为三层：
   - A组（挑战层）：前序知识扎实，代码规范好，可承担拓展任务
   - B组（进阶层）：基本概念掌握，但安全意识或规范性有欠缺
   - C组（基础层）：核心概念仍有薄弱，需要更多引导和脚手架支持
2. 输出每组的共性特征和典型薄弱知识点
3. 给出每组在本讲课堂中的建议关注点
4. 标注需要教师特别关注的"边界学生"（处于两层交界的学生）

## 输出格式
- 分层结果表（学号、姓名、分组、关键指标、分层依据）
- 各组画像摘要（100字以内）
- 教师备课建议（每组1-2条）
```

<div class="info-box amber">
  <span class="icon">📝</span>
  <div><strong>调整建议：</strong>学生数据可从教务系统或在线评测平台导出。分层标准可根据班级整体水平调整阈值。如果班级规模较小（30人以下），也可简化为两层分组。数据维度可根据实际采集情况增减。</div>
</div>

### Prompt 2：个性化实验题目生成

<div class="case-badges">
  <span>一人一题</span>
  <span>差异化教学</span>
  <span>防抄袭</span>
</div>

**使用场景：** 根据学情分层结果，为每位学生生成具有差异化的实验任务。A组接受拓展型任务，B组完成标准型任务，C组获得脚手架型任务，确保"一人一题"降低抄袭风险的同时适配不同能力层次。

```prompt
你是Java Web课程的实验任务设计助手。请根据学生分层信息，为每位学生生成个性化的实验题目。

## 课堂主题
[第N讲主题，例如：用户认证与权限控制]

## 核心知识点
[列出本讲核心知识点，例如：Authentication, Authorization, Session, RBAC, Filter, Password Security]

## 分层要求

### C组（基础层）任务模板
- 提供部分已实现的代码骨架，学生补全关键逻辑
- 明确指出需要修改的文件和方法名
- 给出输入输出示例和验证步骤
- 业务场景：[简单场景，如：图书借阅系统的登录模块]

### B组（进阶层）任务模板
- 提供完整但包含2-3个逻辑缺陷的工程
- 学生需要诊断问题、修复并验证
- 不直接指出缺陷位置，需自行定位
- 业务场景：[中等场景，如：在线商城的用户角色管理]

### A组（挑战层）任务模板
- 仅提供需求描述和接口规范
- 学生需独立设计权限模型并实现完整功能
- 要求包含单元测试和安全审查报告
- 业务场景：[复杂场景，如：多租户SaaS平台的RBAC权限体系]

## 输出要求
1. 每位学生一个独立任务卡片，包含：任务编号、所属分组、业务场景描述、技术要求、提交物清单、评分标准概要
2. 同组内不同学生的业务场景应有差异（不同实体、不同角色名、不同业务规则）
3. 所有任务均须覆盖本讲核心知识点，只是深度和脚手架程度不同
4. 每个任务附带一个"自检清单"供学生提交前自查
```

<div class="info-box amber">
  <span class="icon">📝</span>
  <div><strong>调整建议：</strong>业务场景可从教师预设的场景库中选取，也可让AI随机生成。建议每学期更新场景库以防跨届抄袭。C组的代码骨架需教师预先审核，确保脚手架程度合理。</div>
</div>

### Prompt 3：实验手册与评分标准生成

<div class="case-badges">
  <span>实验手册</span>
  <span>Rubric</span>
  <span>评分标准</span>
</div>

**使用场景：** 课前为本讲实验生成规范的实验指导手册和结构化评分标准（Rubric），确保学生清楚任务要求和评价维度，同时为教师和AI后续评分提供统一口径。

```prompt
你是Java Web课程的教学文档助手。请为本讲实验生成完整的实验手册和评分标准。

## 基本信息
- 实验名称：[例如：用户认证与权限控制实验]
- 对应讲次：[第N讲]
- 实验学时：[2学时 / 4学时]
- 前置知识：[Servlet, JSP, Session, Filter基础]

## 实验手册生成要求
1. 实验目标：列出3-5条可观测的学习目标（使用布鲁姆动词）
2. 实验环境：JDK版本、Tomcat版本、IDE要求、项目结构说明
3. 实验步骤：按阶段划分（环境准备 → 代码审查 → 问题修复 → 功能验证 → 实验报告）
4. 关键代码位置提示：标注需要重点关注的Java类和JSP文件
5. 常见问题FAQ：预判3-5个学生容易遇到的配置或逻辑问题

## 评分标准（Rubric）生成要求
请生成四维评分表，每个维度设置4个等级（优秀/良好/合格/不合格）：

| 维度 | 权重 | 评价内容 |
|------|------|----------|
| 功能实现 | 40% | 认证、授权、会话管理功能是否正确运行 |
| 知识理解 | 25% | 实验报告中对知识点的解释是否准确 |
| 代码规范 | 20% | 命名、注释、异常处理、项目结构 |
| 安全意识 | 15% | 是否关注密码安全、越权防护、会话安全 |

## 输出格式
- 实验手册：Markdown格式，可直接发布到课程网站
- 评分表：表格格式，教师可直接打印使用
- 各分层（A/B/C）的评分标准差异说明
```

<div class="info-box amber">
  <span class="icon">📝</span>
  <div><strong>调整建议：</strong>评分维度权重可根据教学侧重调整。如果课程更强调安全意识，可提高安全维度权重。实验手册生成后建议教师通读一遍，补充校本特色内容（如实验室环境特殊配置）。</div>
</div>

### Prompt 4：代码审查总控提示词

<div class="case-badges">
  <span>教师端</span>
  <span>代码审查</span>
  <span>全局视角</span>
</div>

**使用场景：** 教师在课堂投影端使用，向全班演示AI辅助代码审查的标准流程。覆盖认证、授权、会话管理、视图一致性等全部审查维度，为后续分组审查建立框架。

```prompt
请审查这个 Java Web 用户认证与权限控制模块，重点检查以下方面：

1. 认证流程是否正确（LoginServlet 的用户验证逻辑）
2. Session 的创建、属性保存、注销是否规范
3. 是否存在未授权访问或越权访问风险
4. RoleBasedAuthFilter 是否真正实现了基于角色的访问控制
5. JSP 页面菜单渲染是否与服务端权限控制逻辑一致
6. 密码处理是否存在安全隐患（明文存储、未加密传输等）
7. 是否可能存在固定会话攻击、会话残留或重复提交问题

请按以下格式逐项输出：
- 问题位置（文件名 + 行号或方法名）
- 风险等级（高/中/低）
- 风险说明（一句话概括）
- 对应课程知识点（Authentication / Authorization / Session / RBAC / Filter）
- 修改建议（最小改动方案，不要重写整段代码）
- 建议验证方式（用什么操作可以验证修复是否生效）
```

<div class="info-box amber">
  <span class="icon">📝</span>
  <div><strong>调整建议：</strong>审查维度可根据不同讲次主题调整。例如在数据库相关章节，可替换为SQL注入、事务管理等审查点。输出格式中的"对应课程知识点"确保AI输出始终与教学目标对齐。</div>
</div>

### Prompt 5：A组认证审查提示词

<div class="case-badges">
  <span>A组专用</span>
  <span>Authentication</span>
  <span>Session管理</span>
</div>

**使用场景：** A组（认证组）学生在分组审查环节使用，聚焦 `LoginServlet` 和 `LogoutServlet` 的认证逻辑，重点审查登录状态建立与销毁的完整性。

```prompt
请审查这个 Java Web 登录与登出模块，重点检查以下方面：

1. LoginServlet 中登录成功后是否正确将用户信息（用户名、角色）保存到 Session
2. 登录失败时是否进行了合理的错误反馈，是否避免泄露具体失败原因
3. LogoutServlet 中注销时是否真正调用了 session.invalidate() 清除登录状态
4. 是否存在会话残留问题（登出后 Session 属性未完全清除）
5. 用户重新登录后是否可能出现旧会话数据干扰的问题
6. 是否对登录请求做了基本的输入校验（空值、长度、特殊字符）

审查范围：
- LoginServlet.java（doPost方法）
- LogoutServlet.java（doGet/doPost方法）
- 相关的 web.xml 或注解配置

请按以下格式输出每个发现的问题：
- 问题位置（类名.方法名）
- 风险说明
- 对应知识点（Authentication / Session-Based Login / Logout / Input Validation）
- 修改建议（只给关键改动，不给完整代码）
- 建议验证方式（用什么操作步骤验证）
```

<div class="info-box amber">
  <span class="icon">📝</span>
  <div><strong>调整建议：</strong>如果课程进度已涵盖密码加密（如BCrypt），可在检查项中增加"密码是否经过哈希处理后再比对"。A组适合基础扎实的学生，审查深度可以更高。</div>
</div>

### Prompt 6：B组授权审查提示词

<div class="case-badges">
  <span>B组专用</span>
  <span>Authorization</span>
  <span>RBAC</span>
  <span>Filter</span>
</div>

**使用场景：** B组（授权组）学生在分组审查环节使用，聚焦 `RoleBasedAuthFilter` 的权限过滤逻辑，重点审查基于角色的访问控制是否真正生效。

```prompt
请审查这个 Java Web 角色权限过滤器（RoleBasedAuthFilter），重点检查以下方面：

1. Filter 是否只判断了"用户已登录"而没有判断用户角色
2. 是否正确从 Session 中获取角色信息并与目标资源的权限要求匹配
3. 普通用户（role=USER）是否仍然可以通过直接输入URL访问 /admin/* 路径
4. 未登录用户访问受保护资源时是否被正确拦截并重定向到登录页
5. Filter 的 URL 映射配置（urlPatterns）是否覆盖了所有需要保护的路径
6. 是否存在授权逻辑过宽（放行过多）或判断遗漏（漏检某些路径）的问题

审查范围：
- RoleBasedAuthFilter.java（doFilter方法）
- @WebFilter 注解或 web.xml 中的 filter-mapping 配置

请按以下格式输出每个发现的问题：
- 问题位置（类名.方法名 或 配置文件位置）
- 风险说明
- 对应知识点（Authorization / RBAC / Filter Chain / URL Pattern）
- 修改建议（只给最小改动方案）
- 建议验证方式（用不同角色登录后分别访问什么路径来验证）
```

<div class="info-box amber">
  <span class="icon">📝</span>
  <div><strong>调整建议：</strong>如果项目中有多个Filter形成过滤器链，可增加"过滤器执行顺序是否正确"的检查项。B组侧重逻辑分析能力，适合中等水平学生在有明确审查方向的前提下深入思考。</div>
</div>

### Prompt 7：C组视图一致性审查提示词

<div class="case-badges">
  <span>C组专用</span>
  <span>View Rendering</span>
  <span>前后端一致</span>
</div>

**使用场景：** C组（视图一致性组）学生在分组审查环节使用，聚焦 JSP 页面中的菜单显示逻辑，重点审查前端展示是否与服务端权限控制保持一致。这是学生最容易忽视的安全盲区。

```prompt
请审查这个 Java Web 项目中 JSP 页面的菜单与按钮显示逻辑，重点检查以下方面：

1. 首页是否根据 sessionScope.user 的存在正确切换登录/未登录状态的显示内容
2. "管理后台"入口菜单是否仅在用户角色为 ADMIN 时才显示
3. 前端使用 JSTL/EL 隐藏的菜单是否与服务端 Filter 的权限控制逻辑一致
4. 是否存在"前端隐藏了入口按钮，但用户直接在地址栏输入URL仍可访问后台"的安全漏洞
5. 用户登出后，页面刷新是否正确回退到未登录状态的显示

审查范围：
- index.jsp（导航菜单区域）
- 相关的 JSTL 条件判断标签（c:if / c:choose）
- sessionScope 中的用户属性字段

请按以下格式输出每个发现的问题：
- 问题位置（文件名 + 相关代码段描述）
- 风险说明
- 对应知识点（View Rendering / EL Expression / Session Scope / Authorization Bypass）
- 修改建议（给出JSP标签层面的最小修改方案）
- 建议验证方式（用不同登录状态分别查看页面显示是否正确）
```

<div class="info-box amber">
  <span class="icon">📝</span>
  <div><strong>调整建议：</strong>C组任务相对聚焦于可视化层面，适合基础层学生从"看得见的效果"入手理解安全问题。教师可提前准备不同角色登录后的页面截图作为参照，帮助学生建立预期。</div>
</div>

### Prompt 8：测试验证清单生成提示词

<div class="case-badges">
  <span>验证闭环</span>
  <span>测试清单</span>
  <span>角色路径</span>
</div>

**使用场景：** 学生完成代码修复后，使用此提示词让AI生成结构化的测试验证清单，确保修复不仅解决了原问题，还未引入新的缺陷。这是"诊断-修复-验证"闭环的最后一步。

```prompt
请为这个 Java Web 用户认证与权限控制模块生成一份完整的测试验证清单。

## 测试背景
学生刚刚完成了以下修复：
[简要描述修复内容，例如：
- 修复了 RoleBasedAuthFilter 只判断登录不判断角色的问题
- 修复了 index.jsp 对所有登录用户都显示"管理后台"的问题
- 修复了 LogoutServlet 未调用 session.invalidate() 的问题]

## 清单生成要求
1. 按"未登录 → USER角色 → ADMIN角色 → 登出后"的角色路径组织测试项
2. 每个测试项包含：测试编号、测试目标、操作步骤（具体到URL和操作动作）、预期结果
3. 包含回归测试项：确认修复未破坏其他正常功能
4. 包含边界测试项：Session过期、浏览器后退按钮、并发登录等场景
5. 标注每个测试项对应的知识点

## 输出格式
请输出为表格格式，列包含：编号 | 测试目标 | 操作步骤 | 预期结果 | 对应知识点
清单项数控制在 8-12 项，确保课堂时间内可完成验证。
```

<div class="info-box amber">
  <span class="icon">📝</span>
  <div><strong>调整建议：</strong>修复内容描述需要根据学生实际修改情况填写。如果课堂时间紧张，可以在提示词中加上"请生成最小验证清单（不超过6项）"的约束。清单可打印为纸质任务单分发给各组。</div>
</div>

## 三、课后提示词

课后提示词用于教学反馈与持续改进阶段，帮助教师高效完成个性化学习报告生成和班级共性问题分析，实现数据驱动的精准教学干预。

### Prompt 9：个性化反馈报告生成

<div class="case-badges">
  <span>个性化反馈</span>
  <span>四维评价</span>
  <span>学习建议</span>
</div>

**使用场景：** 课后根据每位学生的实验提交、代码审查记录和课堂表现，由AI生成结构化的个性化反馈报告。报告从功能实现、知识理解、代码规范、安全意识四个维度进行评价，并给出针对性的学习建议。

```prompt
你是Java Web课程的教学反馈助手。请根据以下学生数据生成一份个性化反馈报告。

## 学生信息
- 学号：[学号]
- 姓名：[姓名]
- 所属分组：[A组/B组/C组]
- 本讲主题：[用户认证与权限控制]

## 学生表现数据
- 实验完成情况：[描述功能实现程度，如：完成了登录功能，但Filter未正确判断角色]
- AI审查结果摘要：[AI代码审查发现的主要问题]
- 课堂修复情况：[哪些问题已修复，哪些未完成]
- 测试验证通过率：[如：6项测试通过4项，T2和T4未通过]
- 课堂参与表现：[如：积极提问，能解释修改原因]

## 报告生成要求
请从四个维度生成评价：

1. **功能实现**（40%）：核心功能是否正确运行，缺陷是否修复
2. **知识理解**（25%）：能否区分Authentication和Authorization，能否解释Session作用
3. **代码规范**（20%）：命名规范、异常处理、注释完整性、项目结构
4. **安全意识**（15%）：是否关注越权风险、密码安全、会话管理

## 输出格式
- 总体评级：优秀 / 良好 / 合格 / 需改进
- 四维雷达图数据（每维度0-100分）
- 具体优点（2-3条，具体到代码层面）
- 待改进项（2-3条，具体到知识点层面）
- 个性化学习建议（根据薄弱维度推荐具体学习任务）
- 下次课前预习重点

报告语气要客观、鼓励性强，避免否定性表述，侧重"下一步如何做得更好"。
```

<div class="info-box amber">
  <span class="icon">📝</span>
  <div><strong>调整建议：</strong>学生表现数据可从课堂记录表、Git提交记录和AI审查日志中提取。批量生成时可编写脚本循环调用，每次替换学生信息和表现数据。四维权重可与实验评分标准（Prompt 3）保持一致。</div>
</div>

### Prompt 10：共性问题提取与干预建议

<div class="case-badges">
  <span>班级分析</span>
  <span>共性问题</span>
  <span>教学干预</span>
</div>

**使用场景：** 课后将全班学生的实验数据和AI审查结果汇总输入，让AI提取班级层面的共性薄弱点，并生成针对性的教学干预建议。帮助教师决定下一讲是否需要安排补充讲解或强化练习，实现数据驱动的教学迭代。

```prompt
你是Java Web课程的教学分析助手。请根据全班学生的课堂表现数据，提取共性问题并生成教学干预建议。

## 输入数据
- 课程：Java Web应用开发
- 本讲主题：[用户认证与权限控制]
- 班级人数：[如：78人]
- 数据来源：全班学生的实验提交记录、AI代码审查汇总、测试验证通过率

## 汇总数据
[粘贴全班汇总数据，建议包含以下维度：
- 各测试项通过率（如：T1通过率92%，T2通过率65%，T4通过率58%）
- AI审查发现的问题频次统计（如："Filter未判断角色"出现47次）
- 各分组完成度对比
- 四维评分均值与标准差]

## 分析要求
1. **共性问题提取**：找出全班超过30%学生存在的共性薄弱点，按严重程度排序
2. **知识点缺口分析**：将共性问题映射到课程知识点（Authentication / Authorization / Session / RBAC / Filter / Security），判断哪些知识点教学效果不达预期
3. **分组对比分析**：A/B/C三组在各维度的表现差异，判断分层策略是否合理
4. **干预建议生成**：
   - 需要在下一讲安排补充讲解的知识点
   - 需要设计强化练习的技能项
   - 需要教师个别辅导的"高风险"学生名单
   - 分层策略是否需要调整（如某些学生应升组或降组）
5. **教学迭代建议**：本讲教学设计中哪些环节效果好可保留，哪些需要优化

## 输出格式
- 共性问题TOP5清单（问题描述 + 影响人数 + 对应知识点）
- 知识点掌握热力图数据（知识点 × 掌握程度）
- 教学干预行动清单（按优先级排序，每项含具体操作建议）
- 下一讲教学建议（200字以内）
```

<div class="info-box amber">
  <span class="icon">📝</span>
  <div><strong>调整建议：</strong>如果缺少量化数据，可退化为定性描述输入（如"大部分学生在Filter部分有困难"）。建议每讲课后都执行一次共性分析，形成学期级别的教学效果追踪数据。干预建议可直接导入下一讲的备课计划。</div>
</div>
