## 修改前代码（学生初始提交）

```java
@WebServlet("/logout")
public class LogoutServlet extends HttpServlet {
    @Override
    protected void doGet(HttpServletRequest req, HttpServletResponse resp)
            throws IOException {
        HttpSession session = req.getSession(false);
        if (session != null) {
            session.removeAttribute("user");
        }
        resp.sendRedirect(req.getContextPath() + "/login.jsp");
    }
}
```

### 存在的问题
1. 只移除了`user`属性，Session中其他属性（如`isLoggedIn`、`role`）仍然残留
2. Session本身未销毁，`JSESSIONID`仍然有效
3. 退出后如果其他代码依赖`isLoggedIn`判断，用户仍可能被误认为已登录

## AI审查输出（摘要）

### AI回复关键内容
```
问题位置：doGet方法第5行 session.removeAttribute("user")
风险说明：removeAttribute仅删除单个属性，Session对象本身仍然
存活。如果Session中存储了多个属性（如isLoggedIn、role），这些
属性不会被清除，可能导致"幽灵登录"状态——系统认为用户仍处于
登录状态。
对应知识点：Session生命周期管理、Logout安全
修改建议：使用session.invalidate()替代removeAttribute，
彻底销毁整个Session。
建议验证方式：退出登录后，通过浏览器开发者工具检查JSESSIONID
是否仍然有效，尝试直接访问受保护资源。
```

## 修改后代码（学生最终提交）

```java
@WebServlet("/logout")
public class LogoutServlet extends HttpServlet {
    @Override
    protected void doGet(HttpServletRequest req, HttpServletResponse resp)
            throws IOException {
        HttpSession session = req.getSession(false);
        if (session != null) {
            session.invalidate();
        }
        resp.sendRedirect(req.getContextPath() + "/login.jsp");
    }
}
```

### 改进点
1. 使用`invalidate()`彻底销毁Session，所有属性一并清除
2. `JSESSIONID`失效，后续请求会创建新Session
3. 消除了"幽灵登录"风险
