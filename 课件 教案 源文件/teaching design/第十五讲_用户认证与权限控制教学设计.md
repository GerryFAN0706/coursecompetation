## 一、教学目标

### 知识目标 (Knowledge Objectives)
1. 区分认证（Authentication）与授权（Authorization）的概念与流程。
2. 掌握基于会话（Session-Based）的登录态维护与注销（Logout）机制。
3. 掌握基于角色（RBAC: Role-Based Access Control）的访问控制思路与过滤器（Filter）实现方式。
4. 了解密码安全（Password Security）：摘要与散列（Hash），理解SHA-256原理，了解生产级推荐算法BCrypt/Argon2。

### 能力目标 (Ability Objectives)
1. 能实现登录、登出、会话保存登录用户信息与基本的失败提示。
2. 能编写基于角色的权限过滤器，对`/admin/*`等受限路径实施访问控制。
3. 能在JSP视图层按角色动态渲染菜单。
4. **能使用AI工具辅助代码审查，发现权限控制缺陷并批判性评估AI的建议。**
5. **能设计验证方案，通过多角色测试路径验证系统安全性。**

### 素质目标 (Quality Objectives)
1. 树立"最小权限原则"与"默认拒绝"的安全意识。
2. 形成合规观与责任心：妥善保管用户敏感信息，不在日志中明文打印密码。
3. **培养AI协作素养：理解AI是辅助工具而非替代品，建立"AI建议→人工判断→验证确认"的工作范式。**
4. **[思政融合]** 以"制度先行、权责明晰"比喻RBAC，强调依法治网、守护秩序与公平正义。

## 三、教学重点与难点

### 教学重点
1. 认证（Authentication）与授权（Authorization）的区别及串联实现。
2. 基于角色的权限过滤器（RoleBasedAuthFilter）的核心逻辑。
3. Session安全管理：`invalidate()` vs `removeAttribute()` 的本质区别。
4. **AI辅助代码审查的规范流程：提示词设计→AI输出→批判性评估→验证确认。**

### 教学难点
1. 前端菜单隐藏 ≠ 后端权限控制——理解"纵深防御"思想。
2. 会话残留导致的安全隐患——退出不彻底的连锁后果。
3. **对AI审查结果的批判性判断——AI可能遗漏问题或给出过度建议，学生需要区分"有效建议"与"噪音"。**

## 五、教学流程与时间安排

> 以下流程与录课视频内容一一对应。

### 第一阶段：问题导入——"能登录 ≠ 能安全控制访问"（0-3分钟，3min）

#### 教师活动
1. 投影展示"课程作业管理系统"登录页面，用`admin/admin123`登录，展示正常功能。
2. 用`tom/tom123`（普通用户）登录，演示以下异常行为：
   - tom登录后首页菜单中出现"管理后台"入口（不应出现）。
   - tom点击"管理后台"，成功进入后台页面（不应允许）。
3. 抛出核心问题（投影大字）：**"如果一个普通用户登录后也能进入管理后台，这个系统安全吗？"**

#### 学生活动
- 观察演示，思考系统存在的安全问题。

#### 设计意图
- 用可观察的系统行为制造认知冲突，激发学生诊断动机。
- 将抽象的"认证与授权"概念具象化为可验证的系统bug。

---

### 第二阶段：任务布置与AI使用规则说明（3-5分钟，2min）

#### 教师活动
1. 将学生分为A、B、C三组，每组发放对应的任务卡：
   - **A组（认证组）**：审查`LoginServlet.java`和`LogoutServlet.java`，聚焦Session管理。
   - **B组（授权组）**：审查`RoleBasedAuthFilter.java`，聚焦角色判断逻辑。
   - **C组（视图一致性组）**：审查`index.jsp`菜单部分，聚焦前端显示与后端权限一致性。
2. 明确AI使用规则（投影展示）：
   - AI是"助教"，不是"答案机"。
   - 任务卡中提供了结构化提示词，请按提示词与AI对话。
   - AI的输出是"候选意见"——是否采纳，由你判断。
   - 每组需准备1-2分钟汇报：AI发现了什么？对应什么知识点？你采纳了哪个建议？

#### 学生活动
- 领取任务卡，阅读审查文件与AI提示词。
- 打开IDE，定位到自己负责审查的代码文件。

#### 设计意图
- 结构化任务分配确保每组聚焦不同知识点，课堂汇报时形成互补。
- 提前设立AI使用边界，防止学生直接让AI重写代码。

---

### 第三阶段：AI辅助分组诊断（5-15分钟，10min）

#### 教师活动
1. 巡视各组，观察学生与AI的交互情况。
2. 对常见问题进行个别指导：
   - 如果AI输出过长，引导学生使用"纠偏提示词"压缩为关键3点。
   - 如果AI直接给出整段代码，引导学生追问"只指出需要修改的位置和理由"。
   - 如果学生不理解AI的建议，引导学生使用"知识点对照提示词"。
3. 在教师机上同步演示：使用"教师总控提示词"对完整项目进行AI审查，作为后续对照。

#### 学生活动
1. 将任务卡中的AI提示词复制到AI工具中，附上对应代码文件。
2. 阅读AI输出，记录以下信息：
   - AI发现的问题（位置、描述）
   - AI给出的风险说明
   - 对应的课程知识点
   - AI的修改建议
3. 小组讨论：AI说的对吗？哪些建议值得采纳？

#### AI工具活动
- 接收结构化提示词和代码，输出问题诊断、风险说明、知识点对应、修改建议、验证方式。

#### 设计意图
- **AI的角色是"放大镜"**：帮助学生更快发现代码问题，但发现之后的理解、判断、验证由学生完成。
- 结构化提示词确保AI输出格式一致、与课程知识点对齐，避免天马行空。
- 教师巡视中的纠偏是关键——这是"AI辅助教学"而非"AI替代教学"的体现。

---

### 第四阶段：各组汇报与教师点评（15-25分钟，10min）

#### 教师活动
1. 依次请A、B、C组代表汇报（每组1-2分钟）：
   - A组：AI发现了LogoutServlet只做了`removeAttribute`而非`invalidate`，Session残留导致退出不彻底。
   - B组：AI发现了RoleBasedAuthFilter只检查`isLoggedIn`，未检查角色，普通用户可越权访问后台。
   - C组：AI发现了index.jsp只要登录就显示"管理后台"入口，未按角色区分。
2. 点评每组汇报：
   - 确认AI发现的正确之处。
   - 纠正AI可能遗漏或表述不准确的地方。
   - 补充学生可能忽略的深层问题。
3. 投影展示教师总控AI审查结果，与各组发现对照。

#### 学生活动
- 各组代表汇报，其他同学对照自己的发现。
- 记录其他组的发现（补全自己的知识盲区）。

#### 设计意图
- 三组汇报形成"认证→授权→视图"的完整知识链。
- 教师点评是AI辅助教学的关键环节：AI发现问题，教师判断准确性、补充深度、纠正偏差。
- 学生通过对比"自己的判断 vs AI的建议 vs 教师的点评"，建立三层认知校准。

---

### 第五阶段：知识讲解——对照知识点（25-32分钟，7min）

#### 教师活动
1. 以各组发现的问题为锚点，系统讲解核心知识点：

   **认证 vs 授权（核心区分）：**
   ```
   Authentication → 你是谁？   → LoginServlet验证用户名密码
   Authorization  → 你能做什么？→ RoleBasedAuthFilter检查角色

   核心原则：登录成功 ≠ 拥有所有权限
   ```

2. Session生命周期与安全管理：
   ```
   创建：req.getSession() → 登录成功后保存用户对象
   读取：session.getAttribute("user") → 获取用户信息
   销毁：session.invalidate() → 彻底销毁会话

   陷阱：removeAttribute 只删单个属性，Session 本身仍存活
   ```

3. RBAC模型与Filter实现：
   ```
   角色定义：ADMIN / USER
   Filter拦截：@WebFilter("/admin/*")
   判断逻辑：user != null && "ADMIN".equals(user.getRole())
   拒绝方式：response.sendError(403)
   ```

4. 前端隐藏 vs 后端控制：
   ```
   前端隐藏菜单 = 用户体验优化（UX），不是安全措施
   后端Filter拦截 = 真正的安全边界
   完整方案 = 前端隐藏 + 后端控制
   ```

5. 密码安全：SHA-256演示原理，生产推荐BCrypt/Argon2。

#### 学生活动
- 对照知识点与AI诊断结果，理解每个问题背后的知识点。

#### 设计意图
- 知识讲解不是"从零开始"，而是"基于诊断的系统化"。学生已通过AI辅助诊断对问题有了直觉感知，此时的讲解是将碎片认知组织为结构化知识。

---

### 第六阶段：课堂修复与验证（32-40分钟，8min）

#### 教师活动
1. 带领学生逐一修复三个问题（现场编码，投影展示）：
   - **修复1：RoleBasedAuthFilter** — 将`isLoggedIn`检查改为通过`session.getAttribute("user")`获取User对象，检查`user.getRole()`是否为ADMIN。
   - **修复2：index.jsp** — 将菜单显示条件从`sessionScope.isLoggedIn`改为`sessionScope.user.role == 'ADMIN'`。
   - **修复3：LogoutServlet** — 将`session.removeAttribute("user")`改为`session.invalidate()`。
   - **修复4：LoginServlet** — 删除冗余的`isLoggedIn`/`role`/`displayName`属性存储，只保留`user`对象。
2. 重新部署，使用测试清单逐项验证：

| 编号 | 测试场景 | 操作 | 预期结果 |
|:---|:---|:---|:---|
| T1 | 未登录访问后台 | 直接访问`/admin/panel` | 重定向到登录页 |
| T2 | 普通用户越权 | tom登录后访问`/admin/panel` | 返回403 |
| T3 | 管理员正常访问 | admin登录后访问`/admin/panel` | 正常进入 |
| T4 | 退出后拦截 | 退出后再访问`/admin/panel` | 重定向到登录页 |
| T5 | 菜单一致性 | tom查看首页菜单 | 无"管理后台"入口 |

#### 学生活动
- 跟随教师同步修改代码，在自己的环境中验证。
- 使用测试清单逐项确认修复效果。

#### 设计意图
- 修复过程是"从知到行"的转化——理解了问题和知识点之后，动手修改并验证。
- 测试清单将"验证"制度化，培养学生"改完必测"的工程习惯。

---

### 第七阶段：课堂总结与课后任务（40-45分钟，5min）

#### 教师活动
1. 投影展示三个核心收获：
   - 认证回答"你是谁"，授权回答"你能做什么"。
   - RBAC + Filter = 后端权限控制的基本骨架。
   - AI帮你发现问题，但理解和验证是你的责任。
2. **课程思政**：以RBAC的"制度先行、权责明晰"类比社会治理——任何系统都需要规则和边界，技术人员的责任是构建公平、安全的数字秩序，保护用户权益，遵守法律法规。
3. 布置课后拓展任务：
   - **基础**：将密码哈希从SHA-256升级为BCrypt，对比安全性差异。
   - **进阶**：增加MANAGER角色，实现三级权限（USER/MANAGER/ADMIN）。
   - **挑战**：增强Session安全——防固定会话攻击、设置Session超时。

#### 学生活动
- 记录课堂总结要点与课后任务。

#### 设计意图
- 总结强化"认证 vs 授权"的核心区分，确保知识落地。
- 课程思政自然融入，不生硬说教。
- 分层课后任务满足不同水平学生的学习需求。

## 六、AI应用环节说明

> 本节完整说明本讲中AI工具的使用设计，是人工智能赛道的核心环节。

### 6.1 使用工具

| 工具 | 使用者 | 场景 |
|:---|:---|:---|
| Cursor（内置AI） | 学生 | 分组代码审查，在IDE中直接与AI对话 |
| ChatGPT / Claude | 学生 | 分组代码审查（备选工具） |
| Claude | 教师 | 课堂总控演示，展示AI审查完整项目的输出 |

### 6.2 应用环节

AI工具在本讲的应用集中在**第三阶段（AI辅助分组诊断，5-15分钟）**，贯穿第四阶段（汇报点评）。

### 6.3 设计目的

1. **提升问题发现效率**：传统课堂中，学生需要逐行阅读代码才能发现权限缺陷，耗时且容易遗漏。AI工具可在30秒内扫描代码并指出潜在问题，将课堂时间从"寻找问题"转移到"理解问题"和"解决问题"。
2. **构建批判性思维**：AI的输出并非总是正确或完整的。通过要求学生"判断AI说的对不对"，培养学生对AI建议的批判性评估能力。
3. **对齐课程知识点**：结构化提示词要求AI按"问题位置→风险说明→对应知识点→修改建议→验证方式"的格式输出，确保AI的回答与本讲教学目标对齐。
4. **降低AI滥用风险**：任务卡明确要求学生"使用AI发现问题"而非"让AI写代码"，AI的角色被限定为"诊断助手"。

### 6.4 操作流程

```
第1步：教师分组并发放任务卡（含结构化AI提示词）
         ↓
第2步：学生将提示词 + 对应代码文件输入AI工具
         ↓
第3步：AI输出诊断结果（问题位置、风险、知识点、建议、验证方式）
         ↓
第4步：学生小组讨论 → 判断AI建议是否有效 → 记录采纳/拒绝理由
         ↓
第5步：各组代表汇报 → 教师点评纠偏
         ↓
第6步：教师带领修复 → 使用测试清单验证
```

### 6.5 预期效果

| 维度 | 传统课堂 | AI辅助课堂 |
|:---|:---|:---|
| 问题发现速度 | 学生逐行阅读，10-15分钟可能仍有遗漏 | AI 30秒输出，学生聚焦理解与判断 |
| 知识点覆盖 | 依赖教师逐一讲解 | AI自动对齐知识点，学生主动对照 |
| 学生参与度 | 被动听讲为主 | 主动诊断→判断→汇报→验证 |
| 批判性思维 | 难以训练 | 通过"评估AI建议"自然培养 |
| 课堂效率 | 发现+讲解+修复紧张 | 发现环节压缩，留更多时间给理解与验证 |

### 6.6 防滥用措施

1. **任务限定**：任务卡要求"使用AI审查代码"，不是"让AI重写代码"。AI的角色被明确限定为"诊断助手"。
2. **提示词结构化**：提示词要求AI按固定格式输出（问题位置、风险、知识点、建议、验证方式），而非生成完整代码。
3. **批判性评估要求**：每组汇报必须回答"AI说的对不对？你采纳了哪个建议？为什么？"——强制学生对AI输出进行判断。
4. **教师纠偏机制**：教师巡视中实时干预（AI输出过长→引导压缩；AI给整段代码→引导追问位置和理由），汇报环节进行最终校准。
5. **验证闭环**：AI的建议必须通过测试清单验证才算有效，防止"AI说什么就信什么"。
6. **AI使用反思**：课后任务中包含"反思AI在本次诊断中的作用与局限"，引导学生建立成熟的AI协作观。

## 八、教学方法与手段

### 教学方法
1. **场景驱动 (Scenario-Driven)**：从"普通用户能进管理后台"的可观察缺陷入手，制造认知冲突。
2. **AI辅助诊断 (AI-Assisted Diagnosis)**：学生使用AI工具进行代码审查，教师引导批判性评估。
3. **分组协作 (Group Collaboration)**：A/B/C三组分别聚焦认证/授权/视图，汇报时形成互补。
4. **代码走查 (Code Walkthrough)**：教师带领逐步修复，学生同步操作。
5. **测试驱动验证 (Test-Driven Verification)**：使用测试清单逐项验证修复效果。
6. **双语教学法 (Bilingual Teaching)**：
   - 英文授课，关键术语中英文对照（Authentication, Authorization, RBAC, Session, Filter）。
   - 代码注释与变量命名使用规范英文。

### 教学手段
- IDE（IntelliJ IDEA / Eclipse）
- Tomcat 9+ 服务器
- 浏览器开发者工具（观察Cookie、Session、HTTP状态码）
- **AI工具（Cursor / ChatGPT / Claude）**
- Marp课件投影

## 十、课程思政融入点

### 融入位置
第七阶段（课堂总结，40-45分钟）

### 融入内容
以RBAC的"制度先行、权责明晰"类比社会治理：

1. **技术层面**：认证回答"你是谁"，授权回答"你能做什么"。最小权限、默认拒绝是安全设计的基石。
2. **社会治理层面**：任何系统（技术系统或社会系统）都需要规则和边界。用户分类、权限隔离，保障的是公平与安全。
3. **开发者责任**：作为技术人员，我们编写的每一行权限控制代码，都在守护用户的隐私与权益。依法治网、保护数据安全，是开发者的职业责任。

### 融入方式
- 自然过渡，不生硬说教。
- 从技术知识点（RBAC、最小权限）直接延伸到社会意义（规则、秩序、责任）。
- 结合《个人信息保护法》实施背景，强调开发者的法律义务。

## 十一、课后拓展任务

### 思考题
1. 如何防御"固定会话攻击"（Session Fixation）？登录成功后是否应更换`JSESSIONID`？
2. 如果一个页面既允许ADMIN也允许MANAGER访问，你会如何编排过滤器的判定逻辑与配置？

### 分层实践作业
| 层次 | 任务内容 |
|:---|:---|
| 基础 | 将密码哈希从SHA-256升级为BCrypt（如`BCrypt.checkpw`），对比彩虹表攻击下的安全性差异 |
| 进阶 | 为`/report/*`新增MANAGER角色的访问控制，实现三级权限（USER/MANAGER/ADMIN），并在JSP中新增对应菜单 |
| 挑战 | 增强Session安全：登录成功后重新生成Session ID、设置Session超时、实现"记住我"功能 |
