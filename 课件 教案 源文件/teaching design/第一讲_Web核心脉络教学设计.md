# 第一讲教学设计：Web的核心脉络—HTTP协议与Tomcat的使命 (20分钟)

## 课程信息
- **课程名称**: Java Web应用开发 (Java Web Application Development)
- **教学方式**: 双语教学（英文授课，关键术语中英文对照）
- **教学主题**: Web的核心脉络：HTTP协议与Tomcat的使命
- **授课时长**: 20分钟
- **适用对象**: Java Web开发初学者，对Web开发有初步兴趣的学生

---

## 一、教学目标

### 知识目标
1.  理解HTTP协议（HyperText Transfer Protocol）是Web通信的基石，掌握其请求/响应（Request/Response）模型。
2.  了解HTTP请求和响应的主要组成部分（请求行Request Line/状态行Status Line、头Headers、体Body）。
3.  掌握Tomcat作为Servlet容器（Servlet Container）的核心作用，并了解其基本工作模式。

### 能力目标
1.  能够使用浏览器开发者工具观察和分析基本的HTTP请求与响应报文。
2.  能够独立完成Tomcat的下载、安装和启动。
3.  能够将一个简单的静态HTML页面成功部署到Tomcat服务器并访问。

### 素质目标
1.  建立对Web技术整体架构的宏观认识，理解各组件的协同作用。
2.  培养严谨的、遵循技术标准的工程思维。
3.  **[思政融合]** 通过了解Web技术发展历程和我国网络建设成就，增强学生的国家意识和民族自豪感。

## 二、学情分析

### 知识基础
- **已掌握**：Java语言基础（面向对象编程、集合框架、异常处理），具备基本的编程能力和调试经验
- **待建立**：Web通信的基本原理（HTTP协议、客户端-服务器模型）、服务器软件的概念与使用方法

### 能力水平
- 学生具备Java编程和简单程序调试能力，但尚未接触过网络编程和Web开发领域
- 大部分学生日常使用浏览器浏览网页，但对浏览器与服务器之间的通信机制缺乏认知
- 学生习惯于在IDE中编写和运行单机Java程序，对"部署"概念较为陌生

### AI素养现状
- 学生已具备基本的AI工具使用经验（如ChatGPT问答、文本生成），约80%的学生使用过ChatGPT或类似工具进行日常问答，但使用场景集中在文本润色和知识查询，尚未将AI应用于技术学习场景。具体而言：约60%的学生仅会使用单轮简单提问（如"解释一下XX概念"），不会通过多轮追问深入探索；约20%的学生有过让AI生成代码片段的经历，但仅限于Python课程中的简单脚本，未涉及Web开发领域
- 本讲为课程第一讲（L1），是学生在本课程中的**首次AI技术学习体验**。学生能够向AI提出简单问题（如"什么是HTTP"），但不具备构造结构化技术提示词的能力——例如无法主动在提示词中指定输出格式（"请以报文格式逐行展示"）、约束条件（"仅使用HTTP/1.1协议"）或验证要求（"请标注必需字段与可选字段"）。本讲需通过教师示范，建立"明确目标 + 指定格式 + 要求解释"的提示词构造模板
- 学生对AI输出的信任度普遍偏高，缺乏对AI生成内容（尤其是技术内容）进行正确性验证的意识和方法。前期调研显示：仅15%的学生会在使用AI回答后通过其他资料交叉验证，超过70%的学生直接将AI输出作为最终答案。本讲需通过"AI生成报文 vs DevTools真实报文"的对比环节，引导学生建立"AI输出需验证"的核心习惯，为后续课程奠定"先思考、再验证"的学习模式基础

## 三、教学重点与难点

### 教学重点
-   HTTP请求 (Request) 和响应 (Response) 的基本结构。
-   Servlet容器 (Container) 的概念以及Tomcat在Web应用中的核心角色。
-   Tomcat的安装、核心目录结构及启动/关闭操作。
-   在Tomcat中部署一个简单的Web应用。

### 教学难点
-   将抽象的HTTP协议概念与浏览器和服务器之间看得见的实际交互过程联系起来。
-   理解为什么需要Tomcat这样的容器来运行Java Web程序，而不能像普通Java程序一样直接运行。

## 四、教学内容与时间安排

### **第一部分：问题引入 - 一次网页请求的背后 (3分钟)**

#### 内容要点
1.  **问题驱动 (Problem-Driven):**
    *   以一个学生最熟悉的场景开场："同学们，当我们在浏览器地址栏输入 `http://www.example.edu.cn` 并敲下回车时，屏幕上出现了XXX大学的官网。这个看似简单的动作背后，究竟隐藏了怎样的技术之旅？我们的浏览器是如何"告诉"服务器它想要什么？服务器又是如何"回应"的？"
2.  **高层概述 (High-Level Overview):**
    *   引出客户端（浏览器）和服务器（网站服务器）这两个核心角色。
    *   强调它们之间需要一种"通用语言"进行沟通，这种语言就是 **HTTP协议 (HyperText Transfer Protocol)**。
3.  **思政融合 (Ideological Integration):**
    *   展示数据或图表，简要介绍我国作为全球最大互联网市场的地位和网络基础设施建设的巨大成就（如5G基站数量、光纤网络覆盖等）。
    *   引出学习Web技术不仅是掌握一门技能，更是未来参与"数字中国"、"网络强国"战略建设的基础，激发学生的学习热情和国家自豪感。

#### 教学方法
-   提问互动，激发学生好奇心。
-   结合宏观数据和国家战略，提升学习的格局和意义。

### **第二部分：HTTP协议 - Web的通用语 (7分钟)**

#### 内容要点
1.  **HTTP请求 (HTTP Request / The Request):** 讲解一个请求就像一封信，包含三个核心部分：
    *   **请求行 (Request Line):** `GET /index.html HTTP/1.1` (告诉服务器Server：我想用GET方法Method，获取/index.html这个资源Resource，我遵循HTTP/1.1版本协议Protocol)
    *   **请求头 (Request Headers):** `Host: www.example.edu.cn`, `User-Agent: Chrome/108...` (附加信息，如同信封上的寄件人信息)
    *   **请求体 (Request Body):** (GET请求通常为空，为后面POST请求做铺垫，可比喻为空信件内容)
2.  **HTTP响应 (HTTP Response / The Response):** 讲解服务器的回信也包含三部分：
    *   **状态行 (Status Line):** `HTTP/1.1 200 OK` (告诉浏览器Browser：我用HTTP/1.1回复你，状态码Status Code是200，一切顺利)
    *   **响应头 (Response Headers):** `Content-Type: text/html` (附加信息，告诉浏览器我给你的是一个HTML文件)
    *   **响应体 (Response Body):** 实际的HTML代码 (信件的正文内容)。
3.  **生动演示 (Live Demo):**
    *   打开任一网站（如学校官网），按下F12打开浏览器开发者工具，切换到`Network`（网络）面板。
    *   刷新页面，实时展示捕获到的HTTP请求和响应报文，逐一对应讲解请求行、响应状态码（`200 OK`），以及`Headers`和`Response`内容，将抽象概念可视化。
    *   故意访问一个不存在的URL，演示`404 Not Found`状态码，加深学生理解。

#### 教学方法
-   比喻教学法（信件模型）。
-   现场演示与理论讲解相结合，建立直观认识。

### **第三部分：Tomcat - Web应用的家 (7分钟)**

#### 内容要点
1.  **为什么需要Tomcat?**
    *   提问："我们写的HTML文件，可以直接用浏览器打开。但如果我们的网页内容需要根据当前时间、用户信息等动态变化（即包含Java代码），浏览器能直接执行Java代码吗？" （引导学生回答：不能）
    *   引出结论：我们需要一个特殊的软件，它既能像Web服务器一样接收HTTP请求，又能运行Java代码。这个"超级服务器"就是 **Servlet容器**。
2.  **什么是Tomcat?**
    *   **定义:** Tomcat是Apache基金会下一个开源、免费的轻量级Web服务器和Servlet容器。
    *   **核心使命:** 它的家就是用来装载和管理我们的Java Web应用的。它负责接收HTTP请求，找到并运行我们写的Java程序（Servlet），然后将程序生成的动态内容组合成一个完整的HTTP响应，再发送回浏览器。
3.  **安装与核心目录 (Installation & Core Directories):**
    *   快速演示Tomcat的下载（官网）、解压（绿色免安装）。
    *   讲解核心目录，强调"各司其职"：
        *   `bin`: **启动器** - 存放启动和关闭脚本 (`startup.bat`/`.sh`, `shutdown.bat`/`.sh`)。
        *   `conf`: **配置中心** - 存放服务器配置文件 (`server.xml`)。
        *   `webapps`: **公寓楼** - 存放我们开发的Web应用的地方，每个应用一个文件夹。
        *   `logs`: **日志室** - 记录服务器运行日志，排错必备。
4.  **部署与运行 (Deploy & Run):**
    *   **第一步：创建应用** - 在桌面创建一个 `myapp` 文件夹。
    *   **第二步：编写主页** - 在 `myapp` 文件夹内创建一个 `index.html` 文件，内容为 `<h1>Hello, Tomcat!</h1>`。
    *   **第三步：入住公寓** - 将整个 `myapp` 文件夹复制到Tomcat的 `webapps` 目录下。
    *   **第四步：启动服务** - 进入 `bin` 目录，双击 `startup.bat` 启动Tomcat。
    *   **第五步：验证访问** - 在浏览器访问 `http://localhost:8080/myapp/index.html`，看到 "Hello, Tomcat!" 字样，部署成功！

#### 教学方法
-   设问-解答式教学。
-   分步演示，指令清晰，确保学生能跟上。

### **第四部分：课堂总结与展望 (3分钟)**

#### 总结要点
1.  **通信靠协议**: 浏览器和服务器通过HTTP协议进行"对话"。
2.  **对话有格式**: HTTP通信包含请求和响应两部分，各有固定格式。
3.  **动态靠容器**: Java Web应用不能独立运行，必须住在像Tomcat这样的Servlet容器里。
4.  **部署很简单**: 我们只需将Web应用文件夹放入Tomcat的`webapps`目录即可完成部署。

#### 展望 (承上启下)
-   "今天，我们成功地让Tomcat为我们展示了一个静态页面。但这仅仅是Web世界的冰山一角。真正的Web应用是动态的、可交互的。"
-   "下一节课，我们将亲手编写第一个Servlet程序，让我们的Web应用不再只是一个'展示板'，而是能真正'听懂'用户请求并做出动态回应的智能应用。敬请期待！"

## 五、教学方法与手段

### 教学方法
1. **问题驱动法 (Problem-Driven Method)**: 以实际问题引导学习
2. **讲授法 (Lecturing)**: 系统讲解核心概念和原理
3. **演示法 (Demonstration)**: 现场演示HTTP请求和Tomcat部署
4. **比喻法 (Analogy)**: 用生活化比喻降低理解难度
5. **双语教学法 (Bilingual Teaching)**: 
   - 课堂采用英文授课，关键技术术语提供中英文对照
   - 重要概念首次出现时给出英文表达和中文释义
   - 鼓励学生使用英文术语回答问题
   - 代码注释和变量命名使用规范英文

### 教学手段
-   多媒体课件 (Marp Slides)
-   浏览器开发者工具 (Browser DevTools)
-   Tomcat服务器实例 (Tomcat Server Instance)

## 六、AI应用环节说明

### 6.1 使用工具

| 工具 | 使用者 | 场景 |
|:---|:---|:---|
| ChatGPT / Claude | 教师 | 概念演示阶段：AI生成HTTP请求/响应报文示例，辅助讲解报文结构 |
| ChatGPT / Claude | 学生 | 课堂练习阶段：学生尝试手写HTTP请求报文，再用AI生成对比验证 |

### 6.2 应用环节
AI工具主要应用于**第二部分"HTTP协议—Web的通用语"**（约第4~10分钟）的概念演示阶段。教师在讲解HTTP请求/响应结构后，现场使用AI生成一个完整的HTTP GET请求报文示例，与浏览器开发者工具中捕获的真实报文进行对比，帮助学生直观理解报文结构。随后，学生先尝试手写一个HTTP请求报文，再使用AI生成同一请求的报文，通过对比发现自己的理解偏差。

### 6.3 设计目的
1. **降低抽象概念的理解门槛**：HTTP报文格式对初学者而言较为抽象，AI可以快速生成格式规范的示例，帮助学生建立直观的结构化认知
2. **培养批判性对比能力**：通过"手写 vs AI生成"的对比环节，学生不仅验证自身理解，还学会审视AI输出是否准确、是否遗漏关键字段
3. **建立AI辅助学习的正确模式**：在课程第一讲即引入AI工具，为后续课程中AI辅助编码打下"先思考、再验证"的学习习惯基础

### 6.4 操作流程
```
第1步：教师讲解HTTP请求报文结构（请求行、请求头、请求体），结合板书或课件展示
     ↓
第2步：教师现场向ChatGPT/Claude提问，将AI输出与浏览器DevTools中捕获的真实报文并排对比
     ↓
第3步：学生动手练习——先独立手写一个HTTP GET请求报文（纸上或文本编辑器），
      然后使用AI生成同一请求的报文
     ↓
第4步：学生对比自己的手写版本与AI生成版本，找出差异并在课堂上讨论
      （如：是否遗漏了Host头、HTTP版本号是否正确等）
```

**第2步示例提示词**:
> "请生成一个包含GET请求行、3个常见请求头（Host、User-Agent、Accept）和空请求体的HTTP请求报文示例，目标URL为http://www.example.edu.cn/index.html，协议版本为HTTP/1.1。请逐行解释每部分的作用，并标注哪些字段是必需的、哪些是可选的。"

**第3步学生练习提示词**:
> "我手写了以下HTTP GET请求报文，请帮我检查格式是否正确，并指出遗漏或错误的部分：[粘贴学生手写内容]"

### 6.5 预期效果

| 维度 | 传统课堂 | AI辅助课堂 |
|:---|:---|:---|
| 概念理解速度 | 依赖教师逐行讲解报文，学生被动接受 | AI快速生成多种场景的报文示例，学生主动对比理解 |
| 练习反馈周期 | 学生手写后需等待教师逐一批改 | 学生可即时用AI生成标准答案进行自我校验 |
| 知识内化深度 | 学生记忆报文格式，容易遗忘 | 通过"手写→AI对比→发现差异→修正"的闭环，形成深层理解 |

**可量化指标**：
| 指标 | 传统课堂预期 | AI辅助课堂预期 | 测量方式 |
|:---|:---|:---|:---|
| 课堂随堂测验正确率（报文结构识别） | 55%-65% | 75%-85% | 课堂即时问答统计 |
| 学生独立手写完整HTTP报文的完成率 | 30%-40% | 60%-70% | 课堂练习收集 |
| 课后作业中报文格式错误率 | 35%-45% | 15%-25% | 作业批改统计 |

### 6.6 防滥用措施
1. **任务限定**：AI仅用于生成HTTP报文示例，不允许学生使用AI回答课堂思考题或替代对协议原理的理解
2. **批判性评估**：要求学生在对比环节中明确标注AI生成报文中是否存在不准确之处（如过时的HTTP版本、不恰当的Header字段）
3. **教师审核**：教师在AI演示环节会预先审查AI输出内容，确保示例的准确性和教学适用性
4. **AI使用边界**：AI是辅助者而非替代者，学生必须理解HTTP协议原理后再参考AI输出，课堂评价以学生自主手写的报文为准

## 七、教学评价

### 过程性评价

| 评价维度 | 评价指标 | 评价方式 |
|:---|:---|:---|
| 知识理解 | 能否准确说出HTTP请求报文的三个组成部分（请求行、请求头、请求体） | 课堂提问 |
| AI使用能力 | 能否合理使用AI工具辅助学习，而非依赖AI直接获取答案 | 教师观察 |
| 动手实践 | 能否在指导下正确使用浏览器开发者工具（DevTools）观察和解读HTTP报文 | 现场操作检查 |
| 对比分析 | 能否发现手写报文与AI生成报文之间的差异并给出合理解释 | 课堂讨论发言 |

### 终结性评价

| 评价维度 | 评价指标 |
|:---|:---|
| 协议理解 | 能否用自己的话完整描述一次从浏览器发出请求到接收响应的HTTP通信过程 |
| 部署能力 | 能否独立完成Tomcat安装、启动，并将一个HTML页面部署到webapps目录成功访问 |
| 综合应用 | 能否手写一个基本正确的HTTP GET请求报文（包含请求行和至少两个常见请求头） |

### AI辅助评价
- **AI出题辅助**：教师使用AI快速生成HTTP报文变体题目（如不同Method、不同状态码的报文），覆盖GET/POST/PUT等多种场景，提高随堂测验出题效率
- **自动化比对批改**：学生课后提交的手写报文作业与AI生成的标准报文进行结构化比对，自动标注缺失字段（如遗漏Host头、Content-Type不匹配），辅助教师快速定位共性错误
- **学生AI使用质量分级评估**（L1首次AI体验基准）：
  - **L1-优秀**：能独立构造包含格式要求和约束条件的提示词，且能识别AI输出与DevTools真实报文之间的差异（占比预期15%-20%）
  - **L1-达标**：能在教师模板指导下使用AI生成报文示例，并能发现手写报文与AI版本的1-2处差异（占比预期50%-60%）
  - **L1-待提升**：仅能向AI提出简单问题，无法构造结构化提示词，对比环节中未能发现差异（占比预期20%-30%）

## 八、课后拓展

### 思考题
1.  HTTP的GET请求和POST请求在使用场景上有什么主要区别？
2.  除了Tomcat，你还知道哪些其他的Web服务器或Servlet容器？
3.  `404 Not Found` 状态码是什么意思？你还见过或听说过哪些常见的HTTP状态码？

### 实践作业
1.  在自己的电脑上成功安装并启动Tomcat。
2.  创建一个简单的个人介绍HTML页面（可以包含图片和超链接），将其命名为`aboutme`项目，并成功部署到Tomcat，通过 `http://localhost:8080/aboutme/` 访问。
