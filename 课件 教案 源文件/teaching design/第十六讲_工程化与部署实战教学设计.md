# 第十六讲教学设计：工程化与部署实战 (20分钟)

## 课程信息
- **课程名称**: Java Web应用开发 (Java Web Application Development)
- **教学方式**: 双语教学（英文授课，关键术语中英文对照）
- **教学主题**: 工程化与部署实战——统一异常与日志、全局错误页、JNDI数据源与配置分离、Tomcat打包部署
- **授课时长**: 20分钟
- **适用对象**: 已掌握Servlet/JSP、Filter/Listener、DAO/事务与连接池基础的学生

---

## 一、教学目标

### 知识目标
1.  理解"配置分离（Configuration Separation）、环境解耦（Environment Decoupling）、可观察性（Observability: 日志Logging/错误页Error Page）"的工程化要求。
2.  掌握统一异常（Unified Exception Handling）与统一日志（Unified Logging）的落地方式（过滤器Filter/拦截器Interceptor思路）。
3.  掌握`web.xml`全局错误页（Global Error Page）配置；理解与统一异常处理的配合关系。
4.  掌握在Tomcat中配置JNDI数据源（JNDI DataSource），并在Web应用中通过JNDI查找`DataSource`。
5.  掌握打包WAR（WAR Package）与部署（Deployment）到Tomcat的基本流程。

### 能力目标
1.  能对现有Web应用接入统一日志与统一异常处理，输出规范日志并友好兜底。
2.  能将数据库连接信息从代码中剥离到容器的JNDI资源中，实现环境配置分离。
3.  能完成项目的WAR打包、部署、验证与回滚。

### 素质目标
1.  强化工程规范与上线意识，形成“可运行、可观测、可运维”的完整闭环。
2.  提升问题定位与应急处置能力（错误页兜底、日志追踪）。
3.  **[思政融合]** 强调规则与秩序：日志留痕、责任可追，保障公共系统的公平与安全，体现现代治理能力与法治精神。

- **已具备的AI能力**：(1)能使用结构化Prompt驱动AI生成多文件代码与配置，Prompt中包含技术约束条件的比例从第1讲的不足10%提升至约85%；(2)能对AI生成的代码进行多维度批判性审查（架构合规性、安全漏洞、性能隐患），审查报告平均识别有效问题≥2.5处；(3)能在AI对话中进行多轮追问与方案对比（如第14讲的隔离级别场景分析），具备"提问→质疑→验证"的完整AI协作闭环能力
- 学生已具备对AI输出进行批判性评估的较强能力，能在代码层面识别结构性问题（层间耦合、职责越界），在安全层面识别权限控制缺陷（越权访问、Session残留），但**尚未将AI审查能力延伸到运维配置领域**——XML配置文件的安全审查（密码明文、端口暴露、连接池参数）与代码审查有不同的关注点和审查策略

- 本讲作为课程的收官之作，将引导学生利用Cursor生成部署配置文件，并重点培养对AI生成配置的安全性审查能力——检查密码明文暴露、端口安全、环境变量配置等工程化安全问题，完成AI应用从"代码层"到"运维配置层"的能力闭环，实现课程AI素养培养的终极目标

## 四、教学内容与时间安排

### 第一部分：问题引入——从课堂Demo到可上线应用 (3分钟)

#### 内容要点
1.  展示“硬编码数据库配置、控制台零散日志、异常白屏”的痛点。
2.  目标：配置外置、异常兜底、日志规范、可一键部署。

#### 教学方法
-   反例对比 + 目标导向。

### 第二部分：统一异常与统一日志、全局错误页 (9分钟)

#### 内容要点
1.  统一异常过滤器（核心思路）：
    ```java
    @WebFilter("/*")
    public class GlobalExceptionFilter implements Filter {
        public void doFilter(ServletRequest r, ServletResponse s, FilterChain c)
                throws IOException, ServletException {
            HttpServletRequest req = (HttpServletRequest) r;
            long startedAt = System.currentTimeMillis();
            String uri = req.getRequestURI();
            try {
                c.doFilter(r, s);
            } catch (Throwable ex) {
                // 1) 记录错误日志（最少包含时间/URI/用户/异常栈）
                ex.printStackTrace(); // 生产中请接入标准日志框架
                // 2) 兜底到统一错误页
                ((HttpServletResponse) s).sendRedirect(req.getContextPath() + "/jsp/500.jsp");
            } finally {
                long cost = System.currentTimeMillis() - startedAt;
                System.out.println("[ACCESS] uri=" + uri + " cost=" + cost + "ms");
            }
        }
    }
    ```
2.  全局错误页（`web.xml`）：
    ```xml
    <error-page>
        <error-code>404</error-code>
        <location>/jsp/404.jsp</location>
    </error-page>
    <error-page>
        <error-code>500</error-code>
        <location>/jsp/500.jsp</location>
    </error-page>
    ```
    - 过滤器兜底与错误页不冲突：先由过滤器记录日志与重定向；或由容器转到错误页。
3.  错误页示例（`500.jsp`）：
    ```jsp
    <h2>抱歉，系统繁忙，请稍后再试</h2>
    <p>时间：${pageContext.request.time}</p>
    <p>如持续出现，请联系管理员并提供访问时间与页面地址。</p>
    ```

#### 教学方法
-   代码走查 + 故障演练（制造异常，观察日志与错误页）。

### 第三部分：配置分离与JNDI数据源、WAR打包部署 (6分钟)

#### 内容要点
1.  在Tomcat配置JNDI数据源（`conf/context.xml`或应用`META-INF/context.xml`）：
    ```xml
    <Resource name="jdbc/demoDS" auth="Container" type="javax.sql.DataSource"
              maxTotal="50" maxIdle="5" maxWaitMillis="10000"
              username="root" password="123456"
              driverClassName="com.mysql.cj.jdbc.Driver"
              url="jdbc:mysql://localhost:3306/demo?useSSL=false&serverTimezone=UTC"/>
    ```
2.  应用侧声明引用（`web.xml`）：
    ```xml
    <resource-ref>
        <description>Demo DataSource</description>
        <res-ref-name>jdbc/demoDS</res-ref-name>
        <res-type>javax.sql.DataSource</res-type>
        <res-auth>Container</res-auth>
    </resource-ref>
    ```
3.  代码查找与使用：
    ```java
    Context init = new InitialContext();
    DataSource ds = (DataSource) ((Context) init.lookup("java:comp/env")).lookup("jdbc/demoDS");
    try (Connection conn = ds.getConnection()) {
        // ... 数据库操作
    }
    ```
4.  WAR打包与部署：
    -   IDE/构建工具打包WAR（`myapp.war`）
    -   复制到Tomcat的`webapps/`目录
    -   启动`bin/startup.bat`或`startup.sh`并访问验证

#### 教学方法
-   步骤演示 + 现场联调。

### 第四部分：课堂总结与展望 (2分钟)

#### 总结要点
1.  统一异常 + 错误页：稳定、可观测、可告警。
2.  JNDI数据源：配置与代码解耦，便于多环境切换。
3.  WAR部署：形成可交付产物与标准流程。

#### 展望（承上启下）
-   建议课后将项目整理为“可部署”的最小产品（MVP），并撰写上线手册与回滚预案。

## 六、AI应用环节说明

### 6.1 使用工具

| 工具 | 使用者 | 场景 |
|:---|:---|:---|
| Cursor（AI代码编辑器） | 教师 | 课堂演示：利用Cursor生成Tomcat部署配置文件（server.xml、context.xml）与部署脚本 |
| Cursor（AI代码编辑器） | 学生 | 实战环节：生成项目部署配置后，对AI输出进行安全性审查（密码明文、端口暴露、环境变量等） |

### 6.2 应用环节
AI辅助主要嵌入"第三部分：配置分离与JNDI数据源、WAR打包部署"环节（第12–18分钟）。在教师讲解完统一异常与日志框架后，进入配置分离与部署阶段时引入Cursor：教师先用Cursor演示生成JNDI数据源配置与部署脚本，随后学生使用Cursor为自己的项目生成部署配置，并重点对AI生成的配置文件进行安全性与规范性审查。

### 6.3 设计目的
1. **加速配置文件编写**：Tomcat的server.xml、context.xml、web.xml resource-ref等配置文件语法繁琐、易出错，AI生成可减少配置语法错误，让学生将精力集中在理解配置含义与安全审查上
2. **培养安全意识与安全审查能力**：AI生成的配置文件可能包含密码明文、默认端口、宽松访问策略等安全隐患，引导学生发现并修复这些问题，建立"安全优先"的工程意识
3. **建立"AI辅助运维"的工程化实践**：在实际DevOps流程中，AI辅助生成部署脚本与配置已成为常见实践，学生需要学会审查AI生成的运维产物而非盲目执行

### 6.4 操作流程
```
第1步：教师在Cursor中输入结构化Prompt，演示如何描述部署配置需求
     ↓
第2步：展示AI生成的配置文件，带领学生按安全审查清单逐项检查
     ↓
第3步：学生使用Cursor为自己的项目生成完整部署配置，独立进行安全性审查并修复问题
     ↓
第4步：教师随机抽取2–3组学生的配置文件进行"安全评审"，总结AI生成配置中的常见安全隐患与最佳实践
```

**教师演示提示词**:
> "为以下Java Web项目生成Tomcat JNDI数据源配置（context.xml和web.xml），然后从安全角度审查：密码是否明文存储、连接池参数是否合理、是否配置了SSL连接。项目信息：MySQL 8.0数据库，数据库名demo，需要配置连接池（初始连接5个，最大连接20个，最大等待时间60秒）。请同时生成context.xml中的Resource配置和web.xml中的resource-ref声明，并在每个配置项旁标注安全风险等级。"

**学生实战提示词**:
> "为我的Java Web课程作业管理系统生成完整的Tomcat部署配置，包括：(1)META-INF/context.xml中的JNDI数据源Resource配置（MySQL 8.0，数据库名为[你的数据库名]）；(2)web.xml中的resource-ref声明；(3)全局错误页配置（404和500）。生成后，请逐项标注每个配置的安全风险点（如明文密码、默认端口、缺失SSL等），并给出生产环境的安全加固建议。"

**安全审查检查清单**（学生对照使用）:
- [ ] 数据库密码是否以明文形式写在context.xml中？（应使用环境变量或加密存储）
- [ ] 数据库连接URL是否配置了useSSL=true？（生产环境必须启用SSL）
- [ ] 连接池maxActive/maxTotal参数是否合理？（过大可能耗尽数据库连接，过小可能导致排队）
- [ ] 连接池maxWait是否设置？（缺失会导致无限等待）
- [ ] 是否使用了默认端口3306/8080？（生产环境应修改为非默认端口）
- [ ] 是否配置了validationQuery用于连接健康检查？

### 6.5 预期效果

| 维度 | 传统课堂 | AI辅助课堂 | 可量化指标 |
|:---|:---|:---|:---|
| 配置文件编写效率 | 学生手动编写XML配置，约需8–10分钟，易因语法错误反复调试 | AI生成配置骨架后学生专注审查与修改，约需3–4分钟 | 时间节省率≥50%，XML语法错误率降低≥80% |
| 安全意识培养 | 教师口头强调"不要明文存储密码"，学生缺乏直观感受 | 学生亲自在AI生成的配置中发现安全隐患（如明文密码），印象深刻 | 安全审查清单6项中，学生平均识别≥4项安全问题 |
| 工程化思维 | 侧重"能配置成功"，忽略安全与规范 | 侧重"配置是否安全、是否可维护"，建立生产级意识 | 学生提交的最终配置中消除明文密码、配置SSL连接的比例≥85% |
| 部署成功率 | 首次部署成功率较低，约40-50%，大量时间用于排查配置错误 | AI生成的配置骨架保证基本语法正确，学生聚焦安全加固 | 首次WAR部署成功率≥80% |

**可量化预期指标汇总**：

| 指标名称 | 目标值 | 测量方式 |
|:---|:---|:---|

| 安全问题识别率 | 安全审查清单6项中平均识别≥4项 | 学生提交的安全审查清单评分 |
| 明文密码消除率 | ≥85%的学生最终配置中无明文密码 | AI自动扫描学生提交的context.xml |

| 安全审查清单合格率 | ≥75%（识别出≥3处安全问题并给出修复方案） | 教师批改审查清单 |

### 6.6 防滥用措施
1. **任务限定**：Cursor仅用于生成部署配置文件（XML配置、部署脚本）的初始版本，安全加固与环境适配必须由学生手动完成
2. **批判性评估**：学生必须提交"配置安全审查清单"，列出AI生成配置中发现的至少3个安全问题及修复方案
3. **教师审核**：教师在学生审查过程中巡视，重点检查学生是否识别出密码明文、默认端口、缺失SSL等关键安全问题
4. **AI使用边界**：AI是辅助者而非替代者，学生必须理解JNDI配置原理与Tomcat部署流程后再参考AI输出，不允许将AI生成配置未经审查直接用于部署

### AI辅助评价
- **配置安全自动扫描（L1/L2/L3分级）**：教师将学生提交的部署配置文件（context.xml、web.xml、部署脚本）输入Cursor，让AI进行自动化安全扫描并按风险等级分级——**L1（高危）**：明文密码暴露、默认管理员凭据未修改、未配置SSL连接，每项扣5分；**L2（中危）**：连接池maxActive过大（>100）或过小（<3）、缺少validationQuery健康检查、使用默认端口3306/8080，每项扣3分；**L3（低危）**：配置缺少注释说明、未使用环境变量实现多环境适配、缺少连接超时配置，每项扣1分。生成分级安全问题清单作为辅助评分参考
- **最佳实践差距分析（量化评分）**：利用AI工具对比学生提交的部署配置与生产级最佳实践模板的差异，从三个维度量化评分：安全性（密码加密、SSL、端口，满分40分）、可用性（连接池参数、超时设置、健康检查，满分30分）、可维护性（配置注释、环境变量使用、多环境适配，满分30分），总分100分，≥80分为优秀，≥60分为合格
- **安全审查清单评分**：将学生提交的"配置安全审查清单"输入AI工具，评估学生识别安全问题的准确性与修复方案的可行性，辅助教师实现大班额下的快速一致性评分。评分标准：问题识别准确性（0-40分）、修复方案可行性（0-30分）、安全意识完整性（是否覆盖密码/端口/SSL/连接池四个维度，0-30分）
