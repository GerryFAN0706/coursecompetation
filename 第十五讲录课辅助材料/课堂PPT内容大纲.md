## 第1页：封面 / 问题导入（0-2分钟）

**标题：** 能登录 ≠ 能安全控制访问

**副标题：** 第十五讲 · 用户认证与权限控制

**核心问题（大字）：**
> "如果一个普通用户登录后也能进入管理后台，这个系统安全吗？"

---

## 第2页：教学目标（5-8分钟）

**知识目标：**
1. 理解 Authentication（认证）与 Authorization（授权）的区别
2. 理解 Session-Based Login 机制
3. 理解 RBAC 在 Java Web 中的基本实现
4. 理解 Filter 在权限控制中的作用

**能力目标：**
1. 能分析登录模块中的认证与授权缺陷
2. 能使用 AI 助手进行代码审查与知识点对照
3. 能修改登录、过滤器、视图层代码
4. 能通过不同角色的访问路径验证系统行为

**今天的 AI 助教角色：**
- 帮你发现问题（代码审查）
- 帮你对照知识点
- 帮你生成验证清单
- **不替你写代码，不替你做判断**

## 第6页：知识点对照（26-30分钟）

**核心板书内容（大字展示）：**

```
Authentication  → 你是谁？      → LoginServlet
Authorization   → 你能做什么？   → RoleBasedAuthFilter
Session         → 保存登录状态   → session.setAttribute / invalidate
Filter          → 统一控制入口   → @WebFilter("/admin/*")
RBAC            → 按角色授权     → ADMIN vs USER
```

**关键洞察（高亮）：**
> "如果你只理解登录，不理解授权，你建的是一个'能跑的系统'，不是一个'值得信赖的系统'。"

## 第7页：最小测试清单（35-39分钟）

| 编号 | 测试场景 | 操作 | 预期结果 |
|:---|:---|:---|:---|
| T1 | 未登录访问后台 | 直接访问 `/admin/panel` | 拦截 → 重定向登录页 |
| T2 | 普通用户访问后台 | `tom/USER` 登录后访问 `/admin/panel` | 拒绝 → 403 |
| T3 | 管理员访问后台 | `admin/ADMIN` 登录后访问 `/admin/panel` | 正常进入 |
| T4 | 退出后再访问 | 退出登录后再访问 `/admin/panel` | 拦截 → 重定向登录页 |
| T5 | 菜单一致性 | 检查不同角色的首页菜单 | 仅 ADMIN 显示"管理后台" |

## 第8页：修复结果验证（39-42分钟）

**标题：** 系统是否从"能登录"变成了"能安全控制访问"？

**三条验证路径（配截图或现场演示）：**

1. 未登录 → `/admin/panel` → **被拦截** ✓
2. tom/USER → `/admin/panel` → **403 拒绝** ✓
3. admin/ADMIN → `/admin/panel` → **正常进入** ✓
