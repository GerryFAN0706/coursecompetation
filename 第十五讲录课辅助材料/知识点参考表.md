# 课程知识点参考表

> 第十五讲：用户认证与权限控制

## 核心知识点

| 知识点 | 英文 | 含义 | 本系统中的体现 |
|:---|:---|:---|:---|
| 认证 | Authentication | **你是谁？** 验证用户身份是否合法 | `LoginServlet` 验证用户名密码 |
| 授权 | Authorization | **你能做什么？** 验证用户是否有权限执行某操作 | `RoleBasedAuthFilter` 检查是否为 ADMIN |
| 基于会话的登录 | Session-Based Login | 用 Session 在多次请求间保持用户登录状态 | `session.setAttribute("user", user)` |
| 基于角色的访问控制 | RBAC | 不同角色拥有不同权限，按角色授权 | ADMIN 可访问后台，USER 不可 |
| 过滤器 | Filter | 在请求到达 Servlet 前统一拦截和处理 | `@WebFilter("/admin/*")` 保护后台路径 |
| 密码哈希 | Password Hash | 密码不以明文存储，使用哈希算法保护 | `PasswordUtil.sha256()` |

## 关键区分

### 认证 vs 授权

```
认证（Authentication）：验证 "你是谁"
  → 登录成功 = 认证通过
  → 登录失败 = 认证不通过

授权（Authorization）：验证 "你能做什么"
  → 管理员访问后台 = 授权通过
  → 普通用户访问后台 = 授权不通过

核心原则：登录成功 ≠ 拥有所有权限
```

### 前端隐藏 vs 后端控制

```
前端隐藏菜单 = 界面体验优化（UX）
  → 普通用户看不到"管理后台"按钮
  → 但如果直接输入 URL 仍然能访问，就不是真正的安全

后端权限控制 = 真正的安全边界
  → Filter 在服务端拦截，无论用户从哪里发起请求
  → 即使知道 URL 也无法绕过

结论：前端隐藏 + 后端控制 = 完整的权限方案
```

### Session 生命周期

```
创建：登录成功后 req.getSession() 创建会话
存储：session.setAttribute("user", user) 保存用户信息
读取：session.getAttribute("user") 获取用户信息
销毁：session.invalidate() 彻底销毁会话

注意：
  - removeAttribute 只删除单个属性，Session 本身仍然存在
  - invalidate 销毁整个 Session，所有属性一并清除
  - 安全登出必须使用 invalidate
```

## 本系统的安全原则

1. **默认拒绝**：未登录或无权限时，默认拒绝访问
2. **最小权限**：每个角色只拥有完成其工作所需的最小权限
3. **后端为准**：所有权限判断以服务端为准，前端仅做辅助展示
4. **完整登出**：退出时彻底销毁 Session，不留残余状态
